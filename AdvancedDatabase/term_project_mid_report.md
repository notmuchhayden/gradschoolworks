## 연구 개요
본 연구는 폐쇄망 환경에서 벡터DB 를 활용한 의미 기반 검색 시스템을 구현할 때, 벡터데이터를 기존 RDBMS 안에서 함께 관리하는 방식과 전영 벡터 데이터베이스에 따로 저장하는 방식의 차이를 비교한다. 비교를 통하여 제한적인 상황에서 어떤 구조를 선택하는 것이 더 나은 것인지에 대한 실증 데이터를 제공하는 것을 목적으로 한다.

비교 벡터DB로서 ‘PostgresSQL + pgvector’ 와 ‘PostgreSQL + Qdrant’ 를 선택 하였다. 둘다 벡터DB 로서 의미검색을 지원하지만 결정적인 차이는 pgvector는 PostgreSQL 에 통합되어 원문 데이터와 벡터 데이터가 같은 저장소에 있는지, 아니면 Qdrant 처럼 벡터 데이터가 외부 저장소에 있는지 차이가 있다.

이 차이는 벡터DB의 동기화 문제 처리와 연관이 있을 것으로 예측한다. 아무래도 RDBMS 안에 같이 존재하는 pgvector 쪽이 동기화문제에서 자유롭지 않을까 하는 예상을 하는 것이다.

본 연구는 어느 구조가 더 빠른지를 주로 비교하기보다, 문서 수정 후 원문 데이터와 임베딩 벡터가 서로 맞지 않는 문제가 검색 결과에 어떤 영향을 주는지를 살펴본다. 분리형 구조에서는 원문 수정 후 벡터 갱신이 늦어질 경우 stale embedding 문제가 생길 수 있다. 본 연구에서는 이 상황을 제한적으로 재현하고, 두 구조에서 문서 수정과 벡터 갱신을 관리하는 방식의 차이를 비교한다.

처음에는 전통적인 키워드 검색과 벡터 유사 검색을 비교하는 방향으로 실험을 진행하는 것으로 계획을 하였다. 키워드 검색은 문서 안에서 사용자가 입력한 단어가 정확하게 매칭 되어야 하지만 벡터DB 는 비슷한 의미가 있으면 검색할 수 있는 장점이 있다. 그러나 이런 비교는 이미 여러 논문에서 많이 다루어진 주제이다. 그래서 교수님의 피드백을 반영하여 연구 방향을 벡터DB 의 구현 차이에 의한 동기화 처리로 변경 하였다.

## 관련 연구
본 연구를 하기 전까지 벡터DB 에 대한 아무런 지식이 없었으므로, 벡터 DB에 대한 기본적인 논문을 살펴 보았다. 벡터DB에 대한 조사 논문에서는 벡터DB가 텍스트, 이미지, 음성 같은 비정형 데이터를 벡터 형태로 저장하고 의미가 유사한 데이터를 빠르게 찾기 위한 시스템이라고 설명한다.[1] 벡터DB 는 자체로서는 정보를 저장하는 데이터베이스가 아니라 인덱스 시스템 이라고 할 수 있다. <그림1>에서 처럼 평문을 벡터 임베딩을 통하여 다차원 벡터로 변형하고, 그것을 다양한 유사도 검색 알고리즘을 통해 가장 가까운 유사도를 보이는 데이터를 찾아서 다시 되돌려주는 시스템이다.

<그림1>

실제 데이터를 저장하는 부분은 전통적인 RDBMS 를 사용하여 구축한다. 앞서 벡터 검색을 통해서 얻은 유사도 데이터를 다시 RDBMS 에서 검색하여 정확한 데이터를 검색하여 최종적인 결과물을 사용자에게 돌려 주는 것이다.

따라서 벡터DB 는 전통적인 RDBMS 와 어떻게 조합하여 사용하는 가도 중요한 문제가 된다. 특히 최근의 RDBMS 는 자체적으로 벡터DB 확장을 제공하는 경우도 있기 때문에 과거보다 다양한 선택이 있다고 볼 수 있다.

유명한 RDBMS인 PostresSQL 은 자체적으로 벡터DB 확장을 제공한다. pgvector 를 사용하면 PostgreSQL 안에 벡터를 저장하고, SQL 문법을 사용하여 벡터 거리 계산과 정렬을 수행할 수 있다[2]. 이 방식의 장점은 문서원문, 메타데이터, 벡터를 하나의 PostgreSQL 안에서 관리할 수 있다는 점이다.

반대로 Qdrant 는 전용 벡터 데이터베이스이다. Qdrant 는 데이터를 point 단위로 저장하며, point 에는 id, vector, playload 가 포함된다. 여기서 payload 는 카테고리나 연도 같은 부가 정보이다. Qdrant 는 전용 벡터DB 이므로 payload 에 filtering 을 하여 여러가지 조건을 검색하는 것이 가능하다. 앞서 언급했듯이 Qdrant 는 벡터DB 엔진이고 실제 데이터는 별도의 RDBMS 에 저장된다. 따라서 전통 RDBMS 와 Qdrant 간에는 동기화 문제가 존재한다.

실험을 위하여 앞서 pgvector 와 Qdrant 간의 비교를 한 논문이 있는지 조사 하였다. 하지만 논문은 발견 할수 없었지만 몇가지 벤치마크 결과를 온라인 상에서 찾아볼 수 있었다.

Tiger Data 의 비교 글에서는 PostgreSQL + pgvector + pgvectorscale과 Qdrant를 5 천만 개 임베딩 규모에서 비교하였다[3]. 이 글에서는 Qdrant가 단일 질의 지연 시간과 인덱스 생성 시간에서 장점을 보였고, PostgreSQL 쪽은 높은 recall 조건에서 처리량이 좋고 SQL 및 트랜잭션을 활용할 수 있다는 점을 강조하였다. 이 자료는 학술 논문은 아니지만, 통합형 구조와 전용 벡터DB 구조를 실제 규모에서 비교했다는 점에서 참고할 만하다.

또 다른 초기 실험 보고서에서는 100만 개 OpenAI 임베딩을 사용하여 pgvector와 Qdrant를 비교하였다[4]. 이 결과에서는 Qdrant가 처리량과 검색 정확도 면에서 pgvector보다 좋게 나왔다. 다만 이 실험은 비교적 초기 버전의 pgvector를 기준으로 한 것이고, 이후 pgvector의 HNSW 인덱스나 관련 확장 기능이 개선되었기 때문에 현재 상황을 그대로 대표한다고 보기는 어렵다. 그래도 전용 벡터DB가 순수 벡터 검색 성능에서는 강점을 가질 수 있다는 점을 보여 주는 자료로 볼 수 있다.

최근 실무형 비교 글에서는 데이터 규모가 작거나 중간 정도이고 이미 PostgreSQL을 사용하고 있다면 pgvector가 단순하고 현실적인 선택이 될 수 있다고 설명한다[5]. 반면 벡터 수가 매우 많거나 동시 질의가 많아지면 Qdrant 같은 전용 벡터DB가 더 적합할 수 있다고 본다. 특히 이 글에서는 pgvector의 장점으로 SQL, JOIN, 트랜잭션, 운영 단순성을 들고 있다. 이 내용은 본 연구의 방향과 관련이 있다. 본 연구도 어느 쪽이 무조건 빠른지를 보려는 것이 아니라, 폐쇄망 환경에서 관리하기 쉬운 구조가 무엇인지 확인하려고 하기 때문이다.

정리하면, 기존 자료들은 대체로 Qdrant 같은 전용 벡터DB가 대규모 벡터 검색 성능에서 강점을 가질 수 있고, pgvector는 기존 PostgreSQL과 통합되어 SQL, 트랜잭션, 운영 단순성에서 장점을 가진다고 설명한다. 그러나 실제 문서 검색 환경에서는 단순 검색 속도뿐 아니라 문서 수정 후 임베딩 갱신, 원문과 벡터의 불일치, 동기화 프로그램의 필요성도 중요하다. 따라서 본 연구는 기존 벤치마크 결과를 참고하되, 통합형 구조와 분리형 구조의 데이터 관리 차이를 중심으로 비교하고자 한다.

## 연구의 차별성
이 연구는 벡터 검색 알고리즘을 새로 제안하거나 검색 속도를 최적화하는 연구가 아니다. 또한 단순히 키워드 검색과 벡터 검색의 품질을 비교하는 것도 아니다. 본 연구의 차별점은 벡터 검색을 실제 시스템에 넣을 때 원문 데이터와 벡터 데이터를 어떻게 함께 관리할 것인지에 있다.

전용 벡터DB를 따로 운영하는 방식은 벡터 검색에는 적합해 보이지만, 원문 데이터가 있는 PostgreSQL과 벡터 데이터가 있는 Qdrant 사이의 동기화 문제가 생길 수 있다. 예를 들어 문서 내용은 수정되었는데 Qdrant에 저장된 임베딩은 아직 이전 문서 기준이라면, 검색 결과가 최신 문서의 의미를 제대로 반영하지 못할 수 있다. 본 연구에서는 이런 상태를 stale embedding 또는 의미적 불일치 문제로 보고 실험한다.

반대로 pgvector를 사용하면 원문과 벡터를 같은 PostgreSQL 안에서 관리할 수 있으므로 구조가 단순해진다. 하지만 이 방식이 항상 더 좋다는 뜻은 아니다. 전용 벡터DB를 분리해서 사용하는 방식이 더 적합한 상황도 있을 수 있다. 따라서 본 연구는 두 방식 중 하나를 무조건 정답으로 정하기보다, 폐쇄망처럼 외부 서비스를 쓰기 어렵고 관리 인력이 제한된 환경에서 어떤 장단점이 나타나는지 비교하는 데 의미를 둔다.

## 연구 내용
본 연구에서는 두가지 벡터DB 를 비교하여 동기화 문제를 검증에 집중한다.

**연구 방법**
실험을 위하여 먼저 Docker 를 통하여 각각의 벡터DB 가 동작할 실험 환경을 구성한다.

**구조 A : PostgreSQL + Qdrant 분리형 구조**
- PostgreSQL : 문서 원문, 제목, 메타데이터, 수정 시각, 임제딩 버전 저장
- Qdrant : 임베딩 벡터와 검색에 필요한 payload 저장
- 특징 : 원문 저장소와 벡터 저장소가 분리되어 있음. 이 구조는 문서 수정 후 불일치 상황을 재현하기 위해 필요하다. PostgreSQL의 원문을 먼저 수정하고, Qdrant의 벡터 갱신을 일부러 지연시키면 원문과 벡터가 어긋난 상태를 만들 수 있다. 마치 서로 다른 2대의 PC 에 각자 데이터베이스를 설치하고 동기화가 어긋나는 상황을 재연하기위해 2개의 Docker 컨테이너를 구성하였다.

**구조 B : PostgresSQL + pgvector 통합형 구조**
- PostgreSQL: 문서 원문, 메타데이터, 임베딩 벡터를 같은 DBMS에 저장
- pgvector: PostgreSQL 내부에서 벡터 타입과 벡터 거리 검색 제공
- 특징: 원문과 벡터를 같은 저장소 안에서 관리함. 하나의 Docker 컨테이너 안에 데이터베이스를 구성한다.

**Docker 컨테이너 구성**
postgres : Qdrant 분리형 구조의 원문 및 메타데이터 저장소
qdrant : 전용 벡터DB 검색 저장소
pgvector : pgvector 통합형 구조의 원문, 메타데이터, 벡터 저장소

**Docker Compose 구성**
Docker Compose를 사용하는 이유는 세 컨테이너를 하나의 실험 환경으로 묶어서 관리하기 위해서이다. 이 실험은 PostgreSQL 하나만 실행하는 단순한 실험이 아니라, PostgreSQL 두 개와 Qdrant 하나를 함께 실행해야 한다. Compose 파일을 사용하면 이미지, 포트, 볼륨, 초기화 SQL, healthcheck를 고정할 수 있어 같은 환경을 반복해서 만들기 쉽다.

또한 `docker compose up -d`로 전체 실험 환경을 한 번에 실행하고, `docker compose ps`로 상태를 확인할 수 있다. 동기화 실험은 여러 번 반복해야 하므로, 환경을 쉽게 올리고 내릴 수 있는 구성이 중요하다.

**임베딩 라이브러리 선택**
본 연구에서는 로컬 환경에서 실행 가능하고 계산 비용이 비교적 낮은 sentence-transformers/all-MiniLM-L6-v2 모델을 사용한다. 이 모델은 384차원 문장 임베딩을 생성하므로 10,000건 규모의 문서 실험에 부담이 크지 않으며, 외부 API 없이 폐쇄망 환경에서도 사용할 수 있다. 본 연구의 목적은 임베딩 모델 성능 비교가 아니라 동일한 임베딩을 두 저장 구조에 저장했을 때 동기화 불일치가 어떻게 나타나는지 확인하는 것이므로, 가볍고 재현성이 높은 모델을 선택하였다.

참고로 pgvector 나 Qdrant 는 임베딩 벡터를 저장하고 검색할 수 있는 방법을 제공하고, 독자적으로 임베딩 수행하지는 않는다. 따라서 별도의 임베딩 라이브러리가 필요하다.

**데이터 설계**
공통 문서 필드
필드 : 설명
doc_id : 문서 고유 ID
title : 제목
content : 본문
category : 카테고리
year : 작성 연도
doc_type : 문서 유형
updated_at : 수정 시각
embedding_version : 임베딩 버전

공통 스키마를 사용한 이유는 저장 구조 외의 변수를 줄이기 위해서이다. 두 구조가 서로 다른 필드나 다른 데이터 분포를 사용하면 문서 수정 후 검색 결과 변화가 저장 구조 때문인지 데이터 설계 때문인지 분리하기 어렵다.
embedding_version 필드는 이번 실험에서 가장 중요한 필드이다. Qdrant 분리형 구조에서 PostgreSQL 문서 버전과 Qdrant 의 임베딩 버전이 일시적으로 불일치 할수 있다. 이 불일치를 기준으로 동기화를 측정할 데이터를 얻는다.

## 실험 내용
아직 실험을 진행하지는 않았지만, 구체적인 실험 계획은 다음과 같다.

**문서 수정과 동기화 불일치 실험**
문서 원문이 수정된 뒤 임베딩 벡터 갱신이 늦어질 때, stale embedding 문제가 검색 결과에 어떤 영향을 주는지 확인한다.

**테스트 개요**
- 전체 문서 중 약 10%를 수정한다.
- 예: 10,000건 중 1,000건 수정
- 수정 전후 의미가 달라지도록 본문을 바꾼다.

*예시:*
- 수정 전: "사내 VPN 접속 오류 처리 절차"
- 수정 후: "제로트러스트 인증 실패 대응 절차"

이렇게 수정하면 예전 의미와 새 의미가 분명히 달라진다. 따라서 Qdrant에 예전 벡터가 남아 있을 때 old meaning query로는 검색되고, new meaning query로는 실험 내용 검색되지 않는 상황을 확인할 수 있다.

*Qdrant 분리형 처리 흐름*
1. PostgreSQL 의 원문을 먼저 수정
2. PostgreSQL 의 embedding_version 을 증가
3. Qdrant 의 기존 vector 와 payload 는 한동안 유지
4. 지연 시간 동안 검색을 수행
5. Qdrant payload 의 embedding_version 이 최신인지 확인
6. old meaning query 로 수정 문서가 검색되는지 확인
7. new meaning query 로 수정 문서가 검색되는지 확인
8. 지연 시간이 지난 뒤 Qdrant vector 와 payload 를 갱신
9. 갱신 후 다시 old meaning retrieval 과 new meaning retrieval 을 측정

*pgvector 통합형 처리 흐름*
1. 수정 본문을 준비
2. 새 embedding 을 생성
3. 본문, embedding, embedding_version을 같은 PostgreSQL update 흐름으로 반영
4. 검색 결과를 측정

**측정지표**
- 수정된 문서중 최신 원문과 Qdrant embedding_version 의 불일치 비율
- 이전 의미의 벡터 검색시 수정된 문서가 검색되는 비율
- 문서 수정과 벡터 갱신 처리에 걸린시간
- pgvector 통합형 구조는 본문과 벡터를 같은 update 에 반영하므로 저장후 불일치 상태가 남지 않는 다고 가정한다.

## 목표 및 계획

**남은 연구 과제**
아직 실제 실험은 진행하지 않은 상태이다. 지금까지는 실험 주제를 정하고 실험 환경을 구축하는데 주력 하였다. 개인적으로 SQL 쿼리문을 만드는 것 이상으로 데이터베이스를 다루어 본적이 없어서 실험 주제를 정하는 것도 어려움이 있었다.

벡터DB 의 동기화 문제를 테스트 하기 위해서 docker 를 통해서 가상의 분리된 환경을 만드는 것이 지금까지의 진행 상태이다.

앞으로 실제 데이터를 다운로드 받아서 가공을 하여 테스트 데이터베이스를 만들어야 한다. 테스트 데이터는 앞서 언급 하였듯이 huggingface 에 공개된 영어 뉴스 데이터를 사용하도록 한다.

그리고 아직 측정 지표가 모호 하다. 어떤 사항을 어떻게 테스트 해야 명확하게 동기화 문제를 드러낼 수 있을지 연구가 필요하다.

**최종 목표**
첫 번째 목표는 두 구조의 기본 동작 차이를 정리하는 것이다. 같은 문서와 같은 질의를 사용하여 검색 결과를 비교하고, 각 구조에서 어떤 데이터가 어디에 저장되는지 확인한다.

두 번째 목표는 메타데이터 조건이 붙은 검색에서 구조적 차이를 확인하는 것이다. 실제 업무 검색에서는 카테고리, 연도, 문서 유형과 같은 조건이 함께 사용되기 때문이다.

세 번째 목표는 문서 수정 후 임베딩 갱신 문제를 확인하는 것이다. 특히 Qdrant 처럼 벡터를 따로 저장하는 구조에서는 원문과 벡터가 일시적으로 맞지 않을 수 있다. 본 연구에서는 이 문제를 stale embedding 또는 의미적 불일치로 보고, 검색 결과에 어떤 영향을 주는지 살펴본다.
