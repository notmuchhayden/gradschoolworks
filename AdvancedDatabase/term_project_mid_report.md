## 연구 개요
본 연구는 폐쇄망 환경에서 벡터DB 를 활용한 의미 기반 검색 시스템을 구현할 때,
벡터데이터를 기존 RDBMS 안에서 함께 관리하는 방식과 전영 벡터 데이터베이스에
따로 저장하는 방식의 차이를 비교한다. 비교를 통하여 제한적인 상황에서 어떤 구조를
선택하는 것이 더 나은 것인지에 대한 실증 데이터를 제공하는 것을 목적으로 한다.
비교 벡터DB로서 ‘PostgresSQL + pgvector’ 와 ‘PostgreSQL + Qdrant’ 를 선택
하였다. 둘다 벡터DB 로서 의미검색을 지원하지만 결정적인 차이는 pgvector는
PostgreSQL 에 통합되어 원문 데이터와 벡터 데이터가 같은 저장소에 있는지, 아니면
Qdrant 처럼 벡터 데이터가 외부 저장소에 있는지 차이가 있다.
이 차이는 벡터DB의 동기화 문제 처리와 연관이 있을 것으로 예측한다. 아무래도
RDBMS 안에 같이 존재하는 pgvector 쪽이 동기화문제에서 자유롭지 않을까 하는
예상을 하는 것이다.
연구 개요
본 연구는 어느 구조가 더 빠른지를 주로 비교하기보다, 문서 수정 후 원문 데이터와
임베딩 벡터가 서로 맞지 않는 문제가 검색 결과에 어떤 영향을 주는지를 살펴본다.
분리형 구조에서는 원문 수정 후 벡터 갱신이 늦어질 경우 stale embedding 문제가
생길 수 있다. 본 연구에서는 이 상황을 제한적으로 재현하고, 두 구조에서 문서 수정과
벡터 갱신을 관리하는 방식의 차이를 비교한다.
처음에는 전통적인 키워드 검색과 벡터 유사 검색을 비교하는 방향으로 실험을 진행하는
것으로 계획을 하였다. 키워드 검색은 문서 안에서 사용자가 입력한 단어가 정확하게
매칭 되어야 하지만 벡터DB 는 비슷한 의미가 있으면 검색할 수 있는 장점이 있다.
그러나 이런 비교는 이미 여러 논문에서 많이 다루어진 주제이다. 그래서 교수님의
피드백을 반영하여 연구 방향을 벡터DB 의 구현 차이에 의한 동기화 처리로 변경
하였다.


## 관련 연구

벡터 데이터베이스에 대한 조사 논문에서는 벡터DB가 텍스트, 이미지, 음성 같은 비정형 데이터를 벡터 형태로 저장하고, 유사한 벡터를 빠르게 찾기 위한 시스템이라고 설명한다[1]. 이런 설명을 보면 벡터DB는 단순히 검색 알고리즘 하나만의 문제가 아니라, 데이터를 어떻게 저장하고 관리할 것인가와도 관련이 있다. 실제 업무 문서 검색 시스템을 만든다면 문서 원문, 제목, 작성일, 카테고리 같은 일반 데이터도 필요하고, 검색을 위한 임베딩 벡터도 함께 필요하다. 따라서 벡터를 기존 RDBMS 안에 둘 것인지, 아니면 전용 벡터DB에 따로 둘 것인지가 중요한 선택이 된다.

PostgreSQL에서 벡터 검색을 사용하는 대표적인 방법은 pgvector 확장 기능이다. pgvector를 사용하면 PostgreSQL 안에 벡터 타입을 저장하고, SQL 문법을 사용하여 벡터 거리 계산과 정렬을 수행할 수 있다[2]. 이 방식의 장점은 문서 원문, 메타데이터, 벡터를 하나의 PostgreSQL 안에서 관리할 수 있다는 점이다. 예를 들어 문서의 카테고리나 작성일 조건을 `WHERE`절로 제한하고, 그 안에서 벡터가 가까운 문서를 찾을 수 있다. 또한 기존 PostgreSQL의 트랜잭션과 권한 관리 방식을 그대로 활용할 수 있다는 점도 장점이다.

반대로 Qdrant는 전용 벡터 데이터베이스이다. Qdrant는 데이터를 point 단위로 저장하며, point에는 id, vector, payload가 포함된다. 여기서 vector는 검색에 사용하는 임베딩이고, payload는 카테고리나 연도 같은 부가 정보이다. Qdrant는 벡터 검색과 payload filter를 함께 사용할 수 있으므로, 특정 조건을 만족하는 문서 중에서 유사한 문서를 찾는 방식도 가능하다. 다만 PostgreSQL과 함께 사용할 경우 문서 원문은 PostgreSQL에 있고, 검색용 벡터는 Qdrant에 있는 구조가 되므로 두 시스템 사이의 데이터 동기화가 필요하다.

pgvector와 Qdrant를 직접 비교한 자료들도 있다. Tiger Data의 비교 글에서는 PostgreSQL + pgvector + pgvectorscale과 Qdrant를 5천만 개 임베딩 규모에서 비교하였다[3]. 이 글에서는 Qdrant가 단일 질의 지연 시간과 인덱스 생성 시간에서 장점을 보였고, PostgreSQL 쪽은 높은 recall 조건에서 처리량이 좋고 SQL 및 트랜잭션을 활용할 수 있다는 점을 강조하였다. 이 자료는 학술 논문은 아니지만, 통합형 구조와 전용 벡터DB 구조를 실제 규모에서 비교했다는 점에서 참고할 만하다.

또 다른 초기 실험 보고서에서는 100만 개 OpenAI 임베딩을 사용하여 pgvector와 Qdrant를 비교하였다[4]. 이 결과에서는 Qdrant가 처리량과 검색 정확도 면에서 pgvector보다 좋게 나왔다. 다만 이 실험은 비교적 초기 버전의 pgvector를 기준으로 한 것이고, 이후 pgvector의 HNSW 인덱스나 관련 확장 기능이 개선되었기 때문에 현재 상황을 그대로 대표한다고 보기는 어렵다. 그래도 전용 벡터DB가 순수 벡터 검색 성능에서는 강점을 가질 수 있다는 점을 보여 주는 자료로 볼 수 있다.

최근 실무형 비교 글에서는 데이터 규모가 작거나 중간 정도이고 이미 PostgreSQL을 사용하고 있다면 pgvector가 단순하고 현실적인 선택이 될 수 있다고 설명한다[5]. 반면 벡터 수가 매우 많거나 동시 질의가 많아지면 Qdrant 같은 전용 벡터DB가 더 적합할 수 있다고 본다. 특히 이 글에서는 pgvector의 장점으로 SQL, JOIN, 트랜잭션, 운영 단순성을 들고 있다. 이 내용은 본 연구의 방향과 관련이 있다. 본 연구도 어느 쪽이 무조건 빠른지를 보려는 것이 아니라, 폐쇄망 환경에서 관리하기 쉬운 구조가 무엇인지 확인하려고 하기 때문이다.

정리하면, 기존 자료들은 대체로 Qdrant 같은 전용 벡터DB가 대규모 벡터 검색 성능에서 강점을 가질 수 있고, pgvector는 기존 PostgreSQL과 통합되어 SQL, 트랜잭션, 운영 단순성에서 장점을 가진다고 설명한다. 그러나 실제 문서 검색 환경에서는 단순 검색 속도뿐 아니라 문서 수정 후 임베딩 갱신, 원문과 벡터의 불일치, 동기화 프로그램의 필요성도 중요하다. 따라서 본 연구는 기존 벤치마크 결과를 참고하되, 통합형 구조와 분리형 구조의 데이터 관리 차이를 중심으로 비교하고자 한다.

## 연구의 차별성
이 연구는 벡터 검색 알고리즘을 새로 제안하거나 검색 속도를 최적화하는 연구가 아니다. 또한 단순히 키워드 검색과 벡터 검색의 품질을 비교하는 것도 아니다. 본 연구의 차별점은 벡터 검색을 실제 시스템에 넣을 때 원문 데이터와 벡터 데이터를 어떻게 함께 관리할 것인지에 있다.

전용 벡터DB를 따로 운영하는 방식은 벡터 검색에는 적합해 보이지만, 원문 데이터가 있는 PostgreSQL과 벡터 데이터가 있는 Qdrant 사이의 동기화 문제가 생길 수 있다. 예를 들어 문서 내용은 수정되었는데 Qdrant에 저장된 임베딩은 아직 이전 문서 기준이라면, 검색 결과가 최신 문서의 의미를 제대로 반영하지 못할 수 있다. 본 연구에서는 이런 상태를 stale embedding 또는 의미적 불일치 문제로 보고 실험한다.

반대로 pgvector를 사용하면 원문과 벡터를 같은 PostgreSQL 안에서 관리할 수 있으므로 구조가 단순해진다. 하지만 이 방식이 항상 더 좋다는 뜻은 아니다. 전용 벡터DB를 분리해서 사용하는 방식이 더 적합한 상황도 있을 수 있다. 따라서 본 연구는 두 방식 중 하나를 무조건 정답으로 정하기보다, 폐쇄망처럼 외부 서비스를 쓰기 어렵고 관리 인력이 제한된 환경에서 어떤 장단점이 나타나는지 비교하는 데 의미를 둔다.

## 연구 내용
본 연구에서는 폐쇄망 환경에서 내부 문서 검색 시스템을 만든다고 가정한다. 폐쇄망 환경에서는 외부 클라우드 서비스나 외부 API를 자유롭게 사용하기 어렵고, 시스템을 운영할 수 있는 장비와 인력도 제한될 수 있다. 이런 상황에서는 검색 성능뿐 아니라 시스템 구성이 얼마나 단순한지, 데이터가 서로 맞지 않는 문제가 생기지 않는지도 중요하다.

비교할 구조는 두 가지이다. 첫 번째는 `PostgreSQL + Qdrant` 구조이다. 이 구조에서는 PostgreSQL이 문서 원문, 제목, 카테고리, 작성 연도, 수정 시각 같은 정보를 저장한다. Qdrant는 문서 검색에 필요한 임베딩 벡터를 저장한다. 필요하다면 Qdrant에도 카테고리나 연도 같은 일부 메타데이터를 payload로 함께 저장할 수 있다. 이 방식은 원문 저장소와 벡터 검색 저장소가 분리되어 있다는 특징이 있다.

이 구조의 장점은 벡터 검색을 전용 시스템에 맡길 수 있다는 점이다. Qdrant는 벡터 검색을 목적으로 만들어진 시스템이므로, 벡터 검색 기능 자체는 강점이 있을 것으로 예상된다. 그러나 문서가 수정되면 PostgreSQL의 원문을 수정하는 것만으로 끝나지 않고, 수정된 문서를 다시 임베딩한 뒤 Qdrant의 벡터도 갱신해야 한다. 이 과정이 늦어지거나 실패하면 원문과 벡터가 서로 맞지 않는 상태가 생길 수 있다.

두 번째는 `PostgreSQL + pgvector` 구조이다. 이 구조에서는 문서 원문, 메타데이터, 임베딩 벡터를 모두 PostgreSQL 안에 저장한다. pgvector를 사용하면 PostgreSQL에서 벡터 타입을 사용할 수 있고, SQL 안에서 벡터 거리 계산도 수행할 수 있다. 이 방식의 장점은 관리 대상이 PostgreSQL 하나로 줄어든다는 점이다. 문서와 벡터를 같은 데이터베이스 안에서 처리할 수 있으므로, 동기화 구조가 상대적으로 단순해질 수 있다.

본 연구에서 가장 중점적으로 보려는 부분은 데이터 동기화와 의미적 불일치 문제이다. PostgreSQL과 Qdrant를 분리해서 사용하면 두 시스템을 항상 같은 상태로 유지하는 것이 쉽지 않을 수 있다. 예를 들어 PostgreSQL의 문서 원문은 수정되었지만 Qdrant의 벡터는 아직 갱신되지 않은 경우, 검색은 예전 문서의 의미를 기준으로 이루어질 수 있다. 반대로 Qdrant의 payload에 저장된 카테고리 정보가 PostgreSQL의 최신 메타데이터와 달라지는 경우도 생길 수 있다.

이러한 문제는 단순한 성능 수치만으로는 잘 드러나지 않는다. 하지만 실제 검색 시스템에서는 검색 결과의 신뢰성과 연결된다. 예를 들어 사용자가 최근에 수정된 문서를 찾고 있는데, 검색 시스템이 예전 문서의 의미를 기준으로 결과를 보여 준다면 검색 품질이 떨어질 수 있다. 따라서 본 연구에서는 문서가 수정되었을 때 벡터가 늦게 갱신되는 상황을 일부러 만들고, 그 상태가 검색 결과에 어떤 영향을 주는지 확인한다.

실험 환경은 외부 API를 사용하지 않는 로컬 환경으로 구성한다. 임베딩 모델은 로컬에서 실행 가능한 `sentence-transformers/all-MiniLM-L6-v2`를 사용할 예정이다. 데이터는 뉴스 기사나 기술 문서처럼 여러 카테고리를 가진 텍스트 데이터로 구성한다. 문서 수는 우선 약 10,000건으로 시작하되, 가능하다면 50,000건 이상으로 늘려서 실험해 볼 계획이다. 다만 본 연구의 중심은 대규모 성능 최적화가 아니므로, 데이터 규모 자체보다 같은 조건에서 두 구조를 비교하는 데 더 중점을 둔다.

실험은 크게 세 부분으로 진행한다. 첫 번째는 기본 구조 비교이다. 같은 문서와 같은 임베딩을 두 구조에 저장하고, 검색 질의를 실행해 결과가 어떻게 나오는지 확인한다. 이때 검색 시간도 기록하지만, 성능 최적화를 목표로 하지 않고 두 구조가 같은 기능을 어떤 방식으로 처리하는지 확인하는 용도로 사용한다.

두 번째는 메타데이터 조건이 있는 검색이다. 실제 문서 검색에서는 단순히 의미가 비슷한 문서를 찾는 것뿐 아니라 특정 카테고리, 작성 연도, 문서 유형 같은 조건을 함께 사용하는 경우가 많다. `PostgreSQL + pgvector` 구조에서는 SQL의 `WHERE` 조건과 벡터 정렬을 함께 사용할 수 있고, `PostgreSQL + Qdrant` 구조에서는 Qdrant의 payload filter를 사용한다. 본 연구에서는 조건이 넓은 경우와 좁은 경우를 나누어 두 구조가 메타데이터를 어떻게 관리하고 검색에 사용하는지 비교한다.

세 번째는 문서 수정과 벡터 갱신 실험이다. 전체 문서 중 일부를 수정한 뒤, `PostgreSQL + Qdrant` 구조에서는 벡터 갱신을 일부러 지연시킨다. 예를 들어 0초, 5초, 30초, 120초 정도의 지연 시간을 두고 검색 결과에 예전 벡터가 얼마나 포함되는지 확인한다. `PostgreSQL + pgvector` 구조에서는 원문과 벡터를 같은 PostgreSQL 안에서 함께 갱신하는 방식으로 비교한다.

마지막으로 두 구조의 관리 방식을 표로 정리한다. 비교 항목은 운영해야 하는 저장소 수, 동기화 프로그램 필요 여부, 원문과 벡터가 어긋났을 때 확인해야 하는 항목 등이다. 이 항목들은 성능 수치처럼 정확한 숫자로만 비교하기는 어렵지만, 폐쇄망 환경에서는 실제로 중요한 요소라고 생각한다. 최종적으로 본 연구는 “어느 시스템이 더 빠르다”는 결론보다, “어떤 상황에서 어떤 구조가 더 관리하기 쉬운가”를 정리하는 것을 목표로 한다.

## 실험 내용
실험의 목적은 `PostgreSQL + pgvector` 구조와 `PostgreSQL + Qdrant` 구조를 같은 조건에서 비교하는 것이다. 다만 본 연구에서는 검색 속도 향상이나 알고리즘 최적화를 주된 목표로 삼지 않는다. 대신 원문 데이터와 벡터 데이터가 어디에 저장되는지에 따라 데이터 동기화, 검색 결과의 일관성, 운영 구조가 어떻게 달라지는지를 확인한다.

먼저 두 구조에 같은 문서 데이터를 넣는다. 각 문서는 문서 ID, 제목, 본문, 카테고리, 작성 연도, 문서 유형, 수정 시각을 가진다. 임베딩 모델은 로컬에서 실행 가능한 `all-MiniLM-L6-v2`를 사용한다. `PostgreSQL + pgvector` 구조에서는 문서 정보와 임베딩 벡터를 같은 테이블 또는 관련 테이블에 저장한다. `PostgreSQL + Qdrant` 구조에서는 문서 원문과 기본 정보는 PostgreSQL에 저장하고, Qdrant에는 문서 ID, 임베딩 벡터, 검색에 필요한 일부 payload를 저장한다.

첫 번째 실험은 기본 검색 동작 확인이다. 같은 질의를 두 구조에 입력하여 top-k 검색 결과를 비교한다. 여기서 top-k는 유사도가 높은 상위 k개의 문서를 의미한다. 이 실험에서는 평균 검색 시간도 간단히 기록하지만, 이것을 연구의 중심 결과로 삼지는 않는다. 두 구조가 같은 질의에 대해 어떤 결과를 반환하는지, 그리고 결과를 얻기 위해 어떤 데이터가 어느 저장소에서 사용되는지를 확인하는 것이 더 중요하다.

두 번째 실험은 메타데이터 조건 검색이다. 예를 들어 특정 검색어에 대해 `category = tech_doc` 또는 `year >= 2023` 같은 조건을 함께 적용한다. pgvector 구조에서는 SQL 조건과 벡터 거리 정렬을 함께 사용하고, Qdrant 구조에서는 payload filter를 사용한다. 이 실험에서는 조건이 넓은 경우와 좁은 경우를 나누어 검색 결과와 처리 방식을 비교한다. 이를 통해 벡터 검색과 일반 조건 검색이 결합될 때 두 구조가 어떻게 다른지 확인한다.

세 번째 실험은 문서 수정 후 동기화 문제를 확인하는 것이다. 문서 일부를 수정한 뒤, Qdrant에 저장된 벡터 갱신을 일부러 지연시킨다. 예를 들어 지연 시간이 0초일 때, 5초일 때, 30초일 때, 120초일 때를 나누어 검색을 수행한다. 이때 PostgreSQL의 문서 내용은 최신인데 Qdrant의 벡터는 이전 내용을 기준으로 남아 있는 경우가 발생할 수 있다. 본 연구에서는 이런 상태를 stale embedding 또는 의미적 불일치로 보고, 검색 결과 상위 k개 안에 이런 문서가 얼마나 포함되는지 측정한다.

마지막으로 운영 구조를 간단한 표로 정리한다. 비교 항목은 운영해야 하는 저장소 수, 동기화 프로그램 필요 여부, 데이터 불일치 점검 방법 등이다. 이 항목들은 성능 수치처럼 정확한 숫자로만 비교하기는 어렵지만, 폐쇄망 환경에서는 실제로 중요한 요소라고 생각한다. 최종 보고서에서는 실험 결과와 함께 두 구조의 장단점을 정리하고, 어떤 조건에서 pgvector 방식이 적합하고 어떤 조건에서 Qdrant 방식이 적합한지 판단 기준을 제시할 예정이다.
