# 벡터 저장 구조별 배치 갱신 실험 보고서

## 1. 실험 목적

이 실험은 1,000건의 문서 변경을 벡터 저장소에 반영할 때 배치 크기가 처리 효율과 최신성에 미치는 영향을 비교한다.

비교 구조는 다음 두 가지이다.

- Qdrant 분리형: PostgreSQL에 원문을 저장하고 Qdrant에 벡터를 별도로 저장
- pgvector 통합형: PostgreSQL에 원문과 벡터를 함께 저장

각 구조에 1건, 10건, 100건, 1,000건 배치 조건을 적용했다. 분리형 구조에서 동기화가 필요한 이유는 사전 검증으로만 확인하고, 본 실험은 적절한 배치 크기를 찾는 데 집중했다.

## 3. 데이터와 실험 조건

- 고정 seed로 synthetic 문서 10,000건 생성
- 고정 seed 기반 384차원 mock embedding 사용
- 동일한 문서 중 앞의 1,000건을 수정 대상으로 사용
- 수정 벡터는 측정 전에 생성하여 임베딩 생성 시간을 제외
- 변경 1,000건은 인위적 대기 없이 연속 발생
- 조건별 warmup 1회 후 본 측정 30회 수행
- 각 반복 전 해당 구조의 수정 대상 1,000건을 원래 상태로 복원
- Qdrant upsert는 `wait=True`로 완료를 확인

비교 조건은 다음과 같다.

| 조건 | 구조 | 배치 크기 | 갱신 횟수 |
|---|---|---:|---:|
| Q1 | Qdrant 분리형 | 1 | 1,000 |
| Q10 | Qdrant 분리형 | 10 | 100 |
| Q100 | Qdrant 분리형 | 100 | 10 |
| Q1000 | Qdrant 분리형 | 1,000 | 1 |
| P1 | pgvector 통합형 | 1 | 1,000 |
| P10 | pgvector 통합형 | 10 | 100 |
| P100 | pgvector 통합형 | 100 | 10 |
| P1000 | pgvector 통합형 | 1,000 | 1 |

## 4. 측정 지표

| 지표 | 설명 |
|---|---|
| `total_processing_time_ms` | 첫 변경 시작부터 1,000건의 마지막 반영 완료까지 걸린 시간 |
| `document_visibility_latency_p95_ms` | 문서 변경 이벤트 발생 후 최신 본문과 벡터가 반영되기까지 걸린 시간의 p95 |
| `consistency_error_ratio_after_update` | 실험 완료 후 최신 버전과 일치하지 않는 문서 비율 |

전체 처리시간의 평균과 30회 측정값의 p95를 계산했다. 문서 반영 지연은 각 반복에서 계산된 문서별 p95의 평균을 사용했다.

## 5. 동기화 필요성 사전 검증

PostgreSQL의 원문과 버전만 변경하고 Qdrant를 갱신하지 않은 상태에서 1,000건 전체가 stale 상태가 되는지 확인했다. 검색 확인은 수정 문서 중 100건을 대상으로 수행했다.

| 상태 | stale ratio | old meaning 검색 | new meaning 검색 |
|---|---:|---:|---:|
| Qdrant 동기화 전 | 1.0 | 100 / 100 | 0 / 100 |
| Qdrant 동기화 후 | 0.0 | 0 / 100 | 100 / 100 |

동기화 전에는 Qdrant가 모두 이전 벡터를 사용했고, 동기화 후에는 새 의미가 검색 결과에 반영됐다. 따라서 분리형 구조에 별도 동기화 절차가 필요하다는 사실을 확인했다.

## 7. 결과 해석

### Qdrant 분리형

Q10은 Q1보다 전체 처리시간이 76.69% 감소했다. 문서 반영 지연은 34.848ms에서 57.135ms로 약 1.64배 증가했지만 여전히 100ms 이하였고, 전체 처리시간은 약 18.23초에서 4.25초로 줄었다. 최신성 요구가 높은 환경에서는 Q1보다 훨씬 효율적인 대안이다.

Q100은 Q10보다 전체 처리시간이 46.96% 더 짧았지만 문서 반영 지연은 약 3.96배 증가했다. Q100은 지연이 약 226ms까지 허용되는 환경에서 처리 효율이 높다.

Q1000은 Q100보다 전체 처리시간이 약 14.25% 더 짧았지만, 문서 반영 지연은 약 8.06배 증가했다. 처리시간 차이는 약 321ms에 불과한 반면 문서는 평균 p95 기준 약 1.82초까지 최신 반영을 기다려야 했다.

따라서 Qdrant에서는 일반적인 기본값으로 Q100이 적절하고, 문서 반영 지연을 100ms 이내로 제한해야 한다면 Q10이 적절하다.

### pgvector 통합형

P1은 평균 92.08초로 모든 조건 중 가장 느렸다. 문서 한 건마다 벡터 인덱스 갱신과 transaction commit을 수행하는 비용이 누적된 결과로 해석할 수 있다.

P10은 P1보다 전체 처리시간이 88.79% 감소했다. 문서 반영 지연은 95.859ms에서 108.277ms로 약 1.13배만 증가했지만, 전체 처리시간은 약 92.08초에서 10.32초로 줄었다.

P100은 P10보다 전체 처리시간이 83.30% 더 짧았다. 문서 반영 지연은 108.277ms에서 187.938ms로 약 1.74배 증가했지만, 전체 처리시간은 약 10.32초에서 1.72초로 크게 줄었다.

P1000은 P100보다 전체 처리시간이 약 27.60% 짧았지만, 문서 반영 지연은 약 6.53배 증가했다. 최신성 요구가 전혀 없는 일괄 작업이 아니라면 P100의 이점이 더 크다.

따라서 pgvector에서는 P100이 가장 현실적인 기본 조건이고, 약 100ms 수준의 반영 지연이 반드시 필요할 때만 P10을 고려할 수 있다.

### 같은 배치 크기의 구조 비교

| 배치 크기 | 전체 처리시간 비교 | 문서 반영 지연 비교 |
|---:|---|---|
| 1 | pgvector가 Qdrant보다 5.05배 느림 | pgvector가 2.75배 큼 |
| 10 | pgvector가 Qdrant보다 2.43배 느림 | pgvector가 1.90배 큼 |
| 100 | pgvector가 Qdrant보다 23.54% 빠름 | pgvector가 16.94% 작음 |
| 1,000 | pgvector가 Qdrant보다 35.44% 빠름 | pgvector가 32.75% 작음 |

1건과 10건처럼 작은 배치에서는 pgvector의 반복 transaction 비용이 크게 나타났다. 반면 100건 이상을 한 transaction으로 처리하면 pgvector가 Qdrant 분리형보다 빠르게 나타났다. Qdrant 분리형은 PostgreSQL 원문 갱신과 Qdrant upsert가 모두 필요하므로 배치가 커져도 두 저장소를 순차적으로 처리하는 비용이 남는다.

## 8. 결론

이번 환경에서는 일반적인 처리 효율과 최신성의 균형에는 두 구조 모두 100건 배치가 적합했다. Q10/P10은 100ms 전후의 반영 지연이 필요한 경우 선택할 수 있는 중간 조건이었다.

- Qdrant 권장 기본값: 100건 배치
- Qdrant 최신성 우선값: 10건 배치
- pgvector 권장 기본값: 100건 배치
- pgvector 최신성 우선값: 10건 배치
- 즉시 반영이 절대적으로 중요하면 1건 배치 사용
- 처리량만 중요하고 약 1초 이상의 반영 지연을 허용하면 1,000건 일괄 처리 고려

특히 pgvector에서 문서별 개별 transaction은 피하는 것이 좋다. Qdrant도 10건만 묶어도 Q1 대비 전체 처리시간이 76.69% 감소했으며, 100건으로 늘리면 처리시간이 추가로 감소했다.

실제 운영에서는 건수 기준만 사용하면 변경 빈도가 낮을 때 배치가 채워지지 않을 수 있다. 따라서 다음과 같은 혼합 정책이 적절하다.

```text
10건 또는 100건이 쌓이거나 최대 대기시간이 경과하면 동기화
```

예를 들어 최신성 우선 환경은 `10건 또는 100ms`, 처리 효율 우선 환경은 `100건 또는 1초` 조건을 적용할 수 있다.

## 9. 한계

- synthetic 문서와 mock embedding을 사용했으므로 검색 품질은 평가하지 않았다.
- 1,000건이 연속 발생하는 burst workload만 측정했다.
- 로컬 Windows Docker Desktop WSL2 환경의 결과이므로 서버 Linux 환경과 절대 시간은 다를 수 있다.
- 기본 자원 조건만 측정했으며 CPU와 메모리 제한 조건은 실행하지 않았다.
- 배치 실패, 재시도, 중복 이벤트, 프로세스 중단 복구는 이번 실험에 포함하지 않았다.
- Qdrant는 저장소 간 동기화이고 pgvector는 단일 DB transaction이므로 측정 차이는 제품 자체뿐 아니라 저장 구조 차이도 포함한다.

후속 실험에서는 `10건 또는 100ms`, `100건 또는 1초` 같은 건수·시간 혼합 정책과 실패 후 재시도 전략을 비교할 필요가 있다.
