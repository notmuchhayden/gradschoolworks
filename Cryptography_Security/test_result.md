# Salsa20/8 및 ChaCha8 구현과 실험 결과

## 1. 실험 개요

본 실험에서는 ARX(Add-Rotate-Xor) 구조를 사용하는 Salsa20과 ChaCha 계열 스트림 암호를 C++로 구현하고, 동일한 BMP 이미지에 대한 암호화 및 복호화 정확성과 처리 성능을 비교하였다. 구현 대상은 전체 20라운드 버전이 아니라 실험 계획에서 정한 Salsa20/8과 ChaCha8이다.

소스 파일명과 클래스명은 계획에 따라 `Salsa20`과 `ChaCha20`으로 작성하였으나, 두 클래스의 `Rounds` 값은 모두 8로 설정하였다. 따라서 본 실험에서 측정한 대상은 Salsa20/8과 ChaCha8이다.

## 4. 알고리즘 구현

### 4.1 공통 구조

두 구현은 256비트 키, 64비트 논스, 64비트 블록 카운터를 사용하며 한 번에 64바이트의 키스트림을 생성한다. 내부 상태는 16개의 32비트 워드로 구성되고, 모든 워드는 리틀 엔디언 방식으로 읽고 저장한다.

`generateBlock()` 함수는 지정된 블록 카운터에 해당하는 64바이트 키스트림을 생성한다. `process()` 함수는 입력을 64바이트 블록으로 나누고 각 블록에 대응하는 키스트림과 XOR하여 결과를 생성한다. 스트림 암호의 특성상 같은 함수를 암호문에 다시 적용하면 원문이 복원되므로 암호화와 복호화가 동일한 코드 경로를 사용한다.

또한 처리할 블록 수가 64비트 카운터 범위를 초과하는 경우 예외를 발생시켜 카운터가 다시 0으로 순환하는 것을 방지하였다.

### 4.2 Salsa20/8

Salsa20/8은 상수 4개, 키 워드 8개, 논스 워드 2개, 카운터 워드 2개를 Salsa20 규격에 맞게 4 x 4 상태에 배치한다. quarter-round에서는 덧셈과 XOR 이후 7, 9, 13, 18비트 왼쪽 회전을 수행한다.

한 번의 double-round는 column round와 row round로 구성된다. 이를 4회 수행하여 총 8라운드를 적용한 뒤, 변환된 상태와 초기 상태를 워드 단위로 더해 64바이트 키스트림 블록을 생성한다.

### 4.3 ChaCha8

ChaCha8은 상수, 키, 카운터, 논스를 행 단위로 배치한다. quarter-round는 네 상태 워드를 두 차례 갱신하며 16, 12, 8, 7비트 왼쪽 회전을 사용한다.

한 번의 double-round는 네 개의 column quarter-round와 네 개의 diagonal quarter-round로 구성된다. 이를 4회 수행하여 총 8라운드를 적용하고, 최종 상태와 초기 상태를 더해 키스트림을 생성한다.

## 5. 테스트 프로그램 구성

`TestMain.cpp`는 다음 순서로 실험을 수행한다.

1. BMP 파일을 바이너리 형식으로 읽는다.
2. 두 알고리즘에 동일한 256비트 키와 64비트 논스를 설정한다.
3. 원본 데이터를 암호화하고, 생성된 암호문을 다시 처리하여 복호화한다.
4. 복호화 결과와 원본을 바이트 단위로 비교한다.
5. 암호문이 원본과 다른지 확인한다.
6. 같은 키, 논스, 카운터로 실행한 결과가 동일한지 확인한다.
7. 초기 블록 카운터를 변경했을 때 결과가 달라지는지 확인한다.
8. Salsa20/8과 ChaCha8의 암호문이 서로 다른지 확인한다.
9. ChaCha8의 영 키·영 논스 공식 알려진 벡터와 첫 블록을 비교한다.
10. 각 알고리즘의 암호화와 복호화를 30회 반복하고 평균 시간과 처리량을 계산한다.
11. 암호화 및 복호화 결과를 `results` 디렉터리에 저장한다.

## 6. 빌드 및 실행

다음 명령으로 Release 빌드와 자동 검증을 수행하였다.

```bash
cd Cryptography_Security
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
ctest --test-dir build --output-on-failure
./build/arx_stream_test data_samples/gradient_512x512.bmp 30
```

CMake 빌드는 오류와 경고 없이 완료되었다. CTest에 등록된 `arx_stream_cipher_validation` 테스트도 다음과 같이 통과하였다.

```text
100% tests passed, 0 tests failed out of 1
```

## 8. 성능 측정 결과

786,486바이트 BMP 파일을 대상으로 암호화와 복호화를 각각 30회 수행한 평균 결과는 다음과 같다.

| 알고리즘 | 암호화 평균 시간 | 암호화 처리량 | 복호화 평균 시간 | 복호화 처리량 |
| --- | ---: | ---: | ---: | ---: |
| Salsa20/8 | 2.209 ms | 339.555 MiB/s | 2.911 ms | 257.689 MiB/s |
| ChaCha8 | 1.801 ms | 416.537 MiB/s | 1.762 ms | 425.671 MiB/s |

이번 실행에서는 ChaCha8의 암호화 시간이 Salsa20/8보다 약 18.5% 짧았고, 복호화 시간은 약 39.5% 짧았다. 두 알고리즘의 암호화와 복호화는 같은 XOR 처리 함수를 사용하므로, 암호화와 복호화 사이의 측정 차이는 알고리즘 자체의 비대칭성이 아니라 CPU 주파수 변화, 운영체제 스케줄링, 캐시 상태 등 실행 환경의 영향으로 해석해야 한다.

## 10. 분석 및 결론

Salsa20/8과 ChaCha8은 모두 덧셈, 비트 회전, XOR만 사용하는 ARX 기반 스트림 암호이며 동일한 키 크기, 논스 크기, 블록 크기와 라운드 수로 구현되었다. 두 구현 모두 BMP 파일의 암호화와 복호화, 결정적 출력, 블록 카운터 반영 조건을 만족하였다. ChaCha8은 알려진 공식 벡터와도 일치하여 핵심 블록 함수의 구현 정확성을 추가로 확인하였다.

성능 측정에서는 ChaCha8이 더 높은 처리량을 보였다. ChaCha의 quarter-round는 각 상태 워드를 두 번 갱신하고 column/diagonal 구조를 사용하므로 Salsa20과 연산 수가 유사하면서도 다른 데이터 흐름과 컴파일 결과를 가진다. 이번 환경에서는 이러한 구조가 더 효율적인 실행 결과로 이어진 것으로 판단된다.

다만 본 결과는 단일 CPU, 단일 입력 크기, 스칼라 C++ 구현에서 측정한 값이다. 일반적인 성능 우위를 판단하려면 입력 크기를 다양화하고, 반복 실험의 분산을 기록하며, CPU 주파수를 고정한 상태에서 SIMD 최적화 구현까지 비교해야 한다. 또한 Salsa20/8과 ChaCha8은 축소 라운드 실험용 구현이므로 실제 보안 시스템에서는 충분한 보안 검토 없이 사용해서는 안 된다.
