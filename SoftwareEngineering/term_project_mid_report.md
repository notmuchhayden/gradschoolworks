# 1. 연구 배경 및 목적
본 보고서는 최근 2~3년간 AI 를 활용한 코드 리팩토링 연구를 종합 검토한다. 주요 학술 논문과 산업 사례를 조사하여, LLM을 통한 리팩토링의 유용성을 검증하고자 한다. 또한 직접 실험으로 LLM 을 통한 자동 리팩토링 시도를 통하여, 2026년 현재 상황에서 자동 리팩토링의 가능성과 한계를 규명 하도록 한다.

# 2. 학술 연구 동향
## 2.1 개요
연구를 수행하기 앞서 이미 비슷한 선행 연구가 있는지, 혹은 이미 실용적으로 쓰이고
있는 툴등이 있는지를 조사 하여 그 결과를 정리 한다. 또한 각 선행 사례로 부터 실질적인 실행 아이디어를 수집하고, 실용적인 수행 방법론을 정립하며, 본 연구에서 진행할 실험의 벤치마크 데이터셋 및 평가지표(Metric) 선정의 근거를 확보한다.

## 2.2 자동 리팩토링의 효과
참고 문헌의 논문 “EM-Assist: Safe Automated Extract Method Refactoring with LLMs”[1] 은 실제로 동작하는 제품 수준의 자동 리팩토링 툴을 제작하면서 LLM 을 사용한 리팩토링 툴의 유용성을 검증한다.

EM-Assist는 LLM 통하여 긴 메서드를 Extract Method 를 수행하여 짧은 여러개의 코드로 분할하는 툴이다. 상용 코드 에디터인 IntelliJ 플러그인 형태로 제공이 된다.

특징적인 부분은 LLM 을 통한 리팩토링에 더해 정적분석을 동시에 수행하여, 여러개의 제안을 만들어서 순위화 하는 메커니즘을 제공 한다. 오픈소스에서 실제 수행된 1,752 건의 리팩토링결과, 상위 5개의 제안 내에서 53.4% 의 비율로 제대로된 리팩토링 제안을 하였다고 한다. 이 결과는 기존의 정적분석 도구의 39.4% 대비 큰 개선이다.

또한 개발자 설문에서 94.4%의 긍정적 평가를 얻었다고 주장한다.

## 2.3 LLM-Driven Code Refactoring 의 문제점 고찰
참고 문헌의 “LLM-Driven Code Refactoring: Opportunities and Limitations”[2] 논문은 LLM 에 기반한 코드 리팩토링의 가능성과 한계를 분석한 포지션 페이퍼이다. 따라서 새로운 연구 결과는 없지만, LLM 을 통한 리팩토링을 함에 앞서서 예상될 수 있는 여러가지 문제점을 지적한다.

### 2.3.1 IDE 통합의 어려움과 제한된 맥락 인식 능력
기존의 IDE 는 이미 구문강조, 코드제안, 리팩토링 기능등을 포함하고 있는 경우가 많은데, 이러한 기능에 LLM 에 기반한 리팩토링 기능을 통합하는 것은 여러가지 문제가 있을 수 있다. 대표적으로 LLM 의 느린 속도로 인한 반응성 문제가 있을 수 있다. LLM 은 많은 계산자원을 소비하며 코드 제안과 리팩토링을 위하여 깊은 추론을 사용할 경우 간단한 작업에도 지연이 발생할 수 있음을 지적한다.
또한 리팩토링 작업은 맥락을 인식하며 기존의 코드를 안전하게 변경하는 것이 목표인데, LLM 이 맥락 인식 능력이 떨어질 경우 잘못된 리팩토링을 수행할 가능성이 존재한다.

### 2.3.2 LLM 에서 존재하는 환각 처리의 어려움
기존의 IDE 에 통합된 리팩토링 기능은 제한된 형태의 리팩토링만을 지원하지만, 규칙에서 벗어난 리팩토링은 수행하지 않는다. 하지만 LLM 을 통한 리팩토링은 언제든지 환각을 일으킬 수 있는 위험이 있으며, 이는 예상치 못하는 코드의 동작을 초래할 수 있다. 이는 LLM 이 방대한 코드로 학습을 진행하지만, 리팩토링을 전문적으로 학습하지는 않기 때문이다.

### 2.3.3 LLM의 리팩토링 과정의 목적에 대한 이해 부족
인간 개발자는 코드 스멜 제거, 가독성 향상, 설계 원칙 준수 등 특정 목적을 달성하기 위해 코드를 리팩토링한다. 그러나 LLM은 수정에 대한 명시적인 근거 없이 동작하기 때문에, 종종 프로젝트의 목표와 맞지 않는 변경을 제안한다. 예를 들어, LLM은 코드 구조 단순화에 집중하면서 근본적인 기술 부채를 해결하지 못하거나, 가독성을 개선하는 동시에 새로운 성능 문제를 도입할 수 있다.

### 2.3.4 LLM이 생성한 리팩토링을 검증하기 위한 테스트 케이스의 부족 또는 부재
LLM 은 환각 현상을 격을 수 있기 때문에, 리팩토링이 성공적으로 되었는지를 검증하기 위한 테스트는 필수적이다. 하지만, 테스트 케이스의 부족은 LLM 에 의한 리팩토링의 신뢰성에 의문을 갖게 한다. 현재까지 나온 테스트 케이스들은 자동생성된 단순한 것들이어서 실제 환경에서는 실패할 가능성을 가지고 있다. 이는 결국 인간 검증을 필요로 한다.

## 2.4 LLM 을 통한 리팩토링의 실무 적용
참고문헌 “AI-Assisted Unit Test Writing and Test-Driven Code Refactoring: A Case Study”[3]의 논문은 LLM 을 통하여 자동으로 유닛 테스트를 생성하고, 이에 더해 자동 리팩토링까지 하려는 시도를 실제 상용 시스템에 적용한 사례 연구이다. 이 보고서의 핵심 주제는 리팩토링이므로 논문의 리팩토링의 부분에 집중 하도록 한다.

### 2.4.1 Agent 적용
해당 논문에서는 리팩토링을 하기 위하여 Gemini 2.5 Pro 와 여러 경량 모델을 사용하여 진행 하였음을 지적하고 있다. 하나의 모델로 진행하기 보다는 최근 화제가 되고 있는 Agent 방식을 사용하여 여러 인공지능이 역할을 분담하여 수행하는 방식을 사용하고 있다. 특히 고성능 LLM 은 계획자(Planner) 로 사용하고 경량 모델은 수행자로 사용하여 임무별로 역할을 분리하여 성능 부하를 줄였다.

### 2.4.2 하네스 엔지니어링 (Harness Engineering)
논문의 저자들은 유닛테스트나 리팩토링시 LLM 에게 통일된 결과물을 유도하기 위해 일종의 하네스 엔지니어링을 시도하였다. 하네스 엔지니어링은 다양한 md 파일에 AI 의 행동 제약 사항을 명시하여, AI 가 예상치 못한 임의의 행동을 하지 못하도록 제약하는 방법을 일컫는다.

# 3. 향후 추진계획
앞서 선행 논문들을 통해 자동 리팩토링을 시도 할 때 어떠한 문제점이 있을 수 있는지, 어떤점을 주의해야 하는지 정리 할 수 있었다. 또한 실질적으로 실험을 할 때 어떤 환경을 조성해야 하는지에 대한 실용적인 가이드를 얻을 수 있었다.
다음 단계로 실험을 할 실험계획을 다음과 같이 수립 하였다.

## 3.1 실험 계획

### 3.1.1 모델 선택
최근 상용 AI 모델들은 대부분 유료 모델이므로 토큰의 제한이 있다. 리팩토링 테스트는 많은 토큰이 소모될 것으로 예상 되므로, 이를 감안하여 비교적 한세대 전의 토큰 비용이 저렴한 ChatGPT 4o, Claude Haiku, Gemini 2.5 Pro 모델등을 우선 고려 한다. 시간과 자원이 허락한다면 로컬 LLM 코딩 도구를 사용하여 테스트를 진행하도록 한다.

### 3.1.2 코드 스멜 선정
모든 코드 스멜에 대하여 테스트를 진행하면 가장 좋겠지만, 시간과 자원의 한계로 인하여 2~3개 정도의 가장 빈번하게 사용되는 대표적인 코드 스멜을 선정하도록 한다.

### 3.1.3 데이터 셋
데이터셋은 각 코드 스멜단 20~50 개 사이를 마련하도록 한다. 먼저 github 혹은 리팩토링 관련 온라인 사이트에서 찾아보고, 여의치 않을 경우 LLM 을 사용하여 자동생성된 데이터를 사용하도록 한다.

### 3.1.4 평가 방법
* 기능 정합성/행동 동등성 검증
    * 리팩토링 결과를 평가하는 방법은 먼저 각 데이터셋마다 유닛 테스트 케이스를 작성하여 기본적인 기능적 정합성을 검증 하도록 한다.
    * 유닛 테스트 케이스는 출력 값을 동시에 검증하므로 행동 동등성도 동시에 검증 하도록 한다.
* 리팩토링 품질 측정
    * 리팩토링의 품질 측정은 간단한 맥케이브 복잡도(McCabe Complexity) 를 통하여 측정하도록 한다.
    * 상용 도구인 SonarQube 등을 통하여 복잡도, 중복도의 품질을 측정 하도록 한다.
