# AI 기반 코드 스멜(Code smell) 탐지 및 자동 리팩토링 성능 분석

1. 연구 배경 및 목적
본 보고서는 최근 2~3년간 AI 를 활용한 코드 리팩토링 연구를 종합 검토한다. 주요 학술 논문과 산업 사례를 조사하여, LLM을 통한 리팩토링의 유용성을 검증하고자 한다. 또한 직접 실험으로 LLM 을 통한 자동 리팩토링 시도를 통하여, 2026년 현재 상황에서 자동 리팩토링의 가능성과 한계를 규명 하도록 한다.

2. 학술 연구 동향
2.1 개요
연구를 수행하기 앞서 이미 비슷한 선행 연구가 있는지, 혹은 이미 실용적으로 쓰이고 있는 툴등이 있는지를 조사 하여 그 결과를 정리 한다. 또한 각 선행 사례로 부터 실질적인 실행 아이디어를 수집하고, 실용적인 수행 방법론을 정립하며, 본 연구에서 진행할 실험의 벤치마크 데이터셋 및 평가지표(Metric) 선정의 근거를 확보한다.

2.2 자동 리팩토링의 효과
참고 문헌의 논문 “EM-Assist: Safe Automated Extract Method Refactoring with LLMs”[1] 은 실제로 동작하는 제품 수준의 자동 리팩토링 툴을 제작하면서 LLM 을 사용한 리팩토링 툴의 유용성을 검증한다. 

EM-Assist는 LLM 통하여 긴 메서드를 Extract Method 를 수행하여 짧은 여러개의 코드로 분할하는 툴이다. 상용 코드 에디터인 IntelliJ 플러그인 형태로 제공이 된다. 특징적인 부분은 LLM 을 통한 리팩토링에 더해 정적분석을 동시에 수행하여, 여러개의 제안을 만들어서 순위화 하는 메커니즘을 제공 한다.

오픈소스에서 실제 수행된 1,752 건의 리팩토링결과, 상위 5개의 제안 내에서 53.4% 의 비율로 제대로된 리팩토링 제안을 하였다고 한다. 이 결과는 기존의 정적분석 도구의 39.4% 대비 큰 개선이다. 

또한 개발자 설문에서 94.4%의 긍정적 평가를 얻었다고 주장한다. <그림 1> 에서 해당 논문은 ChatGPT 3.5 를 통하여 Extract Method 를 수행하고 있음을 명시하였다.

2.3 LLM-Driven Code Refactoring 의 문제점 고찰
참고 문헌의 “LLM-Driven Code Refactoring: Opportunities and Limitations”[2] 논문은 LLM 에 기반한 코드 리팩토링의 가능성과 한계를 분석한 포지션 페이퍼이다. 따라서 새로운 연구 결과는 없지만, LLM 을 통한 리팩토링을 함에 앞서서 예상될 수 있는 여러가지 문제점을 지적한다.

2.3.1 IDE 통합의 어려움과 제한된 맥락 인식 능력
기존의 IDE 는 이미 구문강조, 코드제안, 리팩토링 기능등을 포함하고 있는 경우가 많은데, 이러한 기능에 LLM 에 기반한 리팩토링 기능을 통합하는 것은 여러가지 문제가 있을 수 있다. 대표적으로 LLM 의 느린 속도로 인한 반응성 문제가 있을 수 있다.

LLM 은 많은 계산자원을 소비하며 코드 제안과 리팩토링을 위하여 깊은 추론을 사용할 경우 간단한 작업에도 지연이 발생할 수 있음을 지적한다.

또한 리팩토링 작업은 맥락을 인식하며 기존의 코드를 안전하게 변경하는 것이 목표인데,LLM 이 맥락 인식 능력이 떨어질 경우 잘못된 리팩토링을 수행할 가능성이 존재한다.

2.3.2 LLM 에서 존재하는 환각 처리의 어려움
기존의 IDE 에 통합된 리팩토링 기능은 제한된 형태의 리팩토링만을 지원하지만, 규칙에서 벗어난 리팩토링은 수행하지 않는다. 하지만 LLM 을 통한 리팩토링은 언제든지 환각을 일으킬 수 있는 위험이 있으며, 이는 예상치 못하는 코드의 동작을 초래할 수 있다. 이는 LLM 이 방대한 코드로 학습을 진행하지만, 리팩토링을 전문적으로 학습하지는 않기 때문이다.

2.3.3 LLM의 리팩토링 과정의 목적에 대한 이해 부족
인간 개발자는 코드 스멜 제거, 가독성 향상, 설계 원칙 준수 등 특정 목적을 달성하기 위해 코드를 리팩토링한다. 그러나 LLM은 수정에 대한 명시적인 근거 없이 동작하기 때문에, 종종 프로젝트의 목표와 맞지 않는 변경을 제안한다. 예를 들어, LLM은 코드 구조 단순화에 집중하면서 근본적인 기술 부채를 해결하지 못하거나, 가독성을 개선하는 동시에 새로운 성능 문제를 도입할 수 있다.

2.3.4 LLM이 생성한 리팩토링을 검증하기 위한 테스트 케이스의 부족 또는 부재
LLM 은 환각 현상을 격을 수 있기 때문에, 리팩토링이 성공적으로 되었는지를 검증하기 위한 테스트는 필수적이다. 하지만, 테스트 케이스의 부족은 LLM 에 의한 리팩토링의 신뢰성에 의문을 갖게 한다.

현재까지 나온 테스트 케이스들은 자동생성된 단순한 것들이어서 실제 환경에서는 실패할 가능성을 가지고 있다. 이는 결국 인간 검증을 필요로 한다.

2.4 LLM 을 통한 리팩토링의 실무 적용
참고문헌 “AI-Assisted Unit Test Writing and Test-Driven Code Refactoring: A Case Study”[3]의 논문은 LLM 을 통하여 자동으로 유닛 테스트를 생성하고, 이에 더해 자동 리팩토링까지 하려는 시도를 실제 상용 시스템에 적용한 사례 연구이다. 이 보고서의 핵심 주제는 리팩토링이므로 논문의 리팩토링의 부분에 집중 하도록 한다.

2.4.1 Agent 적용
해당 논문에서는 리팩토링을 하기 위하여 Gemini 2.5 Pro 와 여러 경량 모델을 사용하여 진행 하였음을 지적하고 있다. 하나의 모델로 진행하기 보다는 최근 화제가 되고 있는 Agent 방식을 사용하여 여러 인공지능이 역할을 분담하여 수행하는 방식을 사용하고 있다. 특히 고성능 LLM 은 계획자(Planner) 로 사용하고 경량 모델은 수행자로 사용하여 임무별로 역할을 분리하여 성능 부하를 줄였다.

2.4.2 하네스 엔지니어링 (Harness Engineering)
논문의 저자들은 유닛테스트나 리팩토링시 LLM 에게 통일된 결과물을 유도하기 위해 일종의 하네스 엔지니어링을 시도하였다.
하네스 엔지니어링은 다양한 md 파일에 AI 의 행동 제약 사항을 명시하여, AI 가 예상치 못한 임의의 행동을 하지 못하도록 제약하는 방법을 일컫는다.

3. 향후 추진계획
앞서 선행 논문들을 통해 자동 리팩토링을 시도 할 때 어떠한 문제점이 있을 수 있는지, 어떤점을 주의해야 하는지 정리 할 수 있었다.
또한 실질적으로 실험을 할 때 어떤 환경을 조성해야 하는지에 대한 실용적인 가이드를 얻을 수 있었다.
다음 단계로 실험을 할 실험계획을 다음과 같이 수립 하였다.

3.1 실험 계획
본 실험의 목적은 LLM 이 코드 스멜을 포함한 코드를 입력받았을 때, (1) 코드 스멜을 정확히 인식하고, (2) 기능을 유지하면서, (3) 실제로 유지보수성이 개선되는 리팩토링을 생성할 수 있는지를 확인하는 것이다. 따라서 단순히 모델별 정답률을 비교하는 것이 아니라, 리팩토링 결과가 실행 가능한지, 기존 테스트를 통과하는지, 복잡도와 중복도 지표가 개선되는지, 그리고 인간 평가자가 보기에 의미 있는 설계 개선인지까지 함께 측정한다.

3.1.1 연구 질문
본 실험은 다음 네 가지 연구 질문을 중심으로 수행한다.

* RQ1. LLM 은 대표적인 코드 스멜을 포함한 코드에서 적절한 리팩토링 방향을 제안할 수 있는가?
* RQ2. LLM 이 생성한 리팩토링 결과는 기존 기능을 보존하는가?
* RQ3. LLM 리팩토링은 복잡도, 중복도, 코드 크기 등의 정량 지표를 실제로 개선하는가?
* RQ4. 모델 규모와 프롬프트 방식에 따라 리팩토링 품질에는 어떤 차이가 발생하는가?

3.1.2 실험 대상 코드 스멜 선정
실험 범위를 과도하게 넓히면 데이터셋 준비와 인간 평가가 어려워지므로, 본 연구에서는 다음 세 가지 코드 스멜을 우선 대상으로 한다.

* 긴 메서드(Long Method): 하나의 함수 또는 메서드가 여러 책임을 동시에 수행하여 이해와 테스트가 어려운 경우이다. 주요 리팩토링 기법은 Extract Method, 변수명 개선, 조건문 분리이다.
* 중복 코드(Duplicated Code): 유사한 로직이 여러 함수나 클래스에 반복되어 수정 비용이 증가하는 경우이다. 주요 리팩토링 기법은 공통 함수 추출, 템플릿 메서드화, 파라미터화이다.
* 큰 클래스(Large Class) 또는 데이터 클래스(Data Class): 하나의 클래스가 과도한 필드와 메서드를 포함하거나, 행위 없이 데이터만 보유하는 경우이다. 주요 리팩토링 기법은 Extract Class, 책임 분리, 캡슐화 강화이다.

선정 기준은 소프트웨어공학 교재와 리팩토링 문헌에서 자주 다루어지는 대표성, 자동 정적분석 지표로 측정 가능한 정도, 그리고 짧은 텀프로젝트 기간 안에 사람이 결과를 검토할 수 있는 현실성이다. 시간이 부족할 경우 긴 메서드와 중복 코드 두 유형으로 범위를 축소한다.

3.1.3 데이터셋 구성
실험 언어는 Python 으로 통일한다. Python 은 실행 환경 구성이 비교적 단순하고, pytest, radon, ruff, pylint 등 정적분석 도구를 쉽게 사용할 수 있으므로 제한된 기간의 텀프로젝트에 적합하다.

데이터셋은 코드 스멜 유형별 10개, 총 30개 샘플을 기본 목표로 한다. 각 샘플은 다음 요소를 포함한다.

* 원본 코드 파일 1개
* 코드 스멜 유형 라벨 1개
* 원본 코드의 기대 동작을 검증하는 pytest 테스트 파일 1개
* 샘플 난이도 라벨: easy, medium, hard
* 사람이 작성한 기준 리팩토링 방향 또는 평가 메모

샘플은 우선 공개 예제, 리팩토링 교재의 예시를 변형한 코드, 직접 작성한 소형 프로그램을 사용한다. LLM 으로 생성한 코드는 데이터 부족 시 보조적으로 사용하되, 실험에 포함하기 전에 사람이 코드 스멜 존재 여부와 테스트 타당성을 검토한다. 데이터셋 규모는 최종 일정에 따라 유형별 8~10개로 조정할 수 있다.

3.1.4 비교 대상 모델 및 실험 조건
모델은 실험 시점에 사용 가능한 상용 LLM 중 코딩 성능이 알려진 모델 2~3개를 선정한다. 기본 비교 대상은 ChatGPT 계열, Claude 계열, Gemini 계열 중 하나씩 선택하는 방식으로 구성한다. 비용과 사용량 제한 때문에 모든 모델에 동일한 수의 샘플과 동일한 프롬프트를 적용하며, 각 모델별 실험 횟수도 동일하게 맞춘다.

추가로, 가능하다면 로컬 또는 오픈소스 코딩 모델 1개를 포함하여 상용 모델과 비교한다. 다만 로컬 모델은 GPU 자원과 설치 시간이 많이 들 수 있으므로 필수 조건이 아니라 확장 실험으로 둔다.

각 모델은 다음 두 가지 프롬프트 조건에서 실험한다.

* 기본 프롬프트: 코드와 리팩토링 요청만 제공한다. 예: "다음 코드의 기능을 유지하면서 코드 스멜을 제거하도록 리팩토링하라."
* 제약 프롬프트: 테스트 통과, 공개 API 유지, 불필요한 기능 추가 금지, 변경 요약 작성 등 하네스 엔지니어링 제약을 포함한다.

이 비교를 통해 단순한 모델 성능뿐 아니라 프롬프트 제약이 리팩토링 안정성에 미치는 영향도 확인한다.

3.1.5 실험 절차
각 샘플에 대해 다음 절차를 동일하게 적용한다.

1. 원본 코드와 테스트 케이스를 준비하고, 원본 코드가 모든 테스트를 통과하는지 확인한다.
2. 원본 코드의 정적 지표를 측정한다. 측정 항목은 LOC, cyclomatic complexity, maintainability index, 중복 코드 비율이다.
3. 동일한 코드와 동일한 프롬프트를 각 LLM 에 입력한다.
4. LLM 이 생성한 결과를 별도 브랜치 또는 별도 디렉터리에 저장한다.
5. 생성 코드에 대해 포맷팅과 최소한의 실행 환경 정리를 수행하되, 사람이 임의로 로직을 수정하지 않는다.
6. 기존 pytest 테스트를 실행하여 기능 보존 여부를 확인한다.
7. 리팩토링 후 정적 지표를 다시 측정한다.
8. 인간 평가자가 코드 스멜 제거 여부와 유지보수성 개선 여부를 루브릭에 따라 평가한다.
9. 필요한 경우 LLM 에게 자기 평가를 수행하게 하여 인간 평가와 비교한다.

실험 과정에서 LLM 이 기존 함수명 또는 클래스명을 임의로 변경하여 테스트가 실패하는 경우가 예상된다. 이 경우 공개 API 유지 조건을 어긴 것으로 기록한다. 단순 import 경로 문제나 코드 블록 마크다운 제거처럼 실험 결과의 본질과 무관한 형식 문제는 별도로 기록하고 최소 수정 후 재실행한다.

3.1.6 평가 지표
평가 지표는 자동 측정 지표와 인간 평가 지표로 나누어 구성한다.

자동 측정 지표는 다음과 같다.

* 테스트 통과율: 리팩토링 후 기존 pytest 테스트를 모두 통과한 샘플의 비율
* 실행 가능률: 문법 오류나 import 오류 없이 실행 가능한 결과의 비율
* 복잡도 변화율: 리팩토링 전후 cyclomatic complexity 의 변화
* 유지보수성 지수 변화: radon 등의 도구가 제공하는 maintainability index 변화
* 코드 크기 변화: LOC 변화량
* 중복도 변화: 동일하거나 유사한 코드 블록의 감소 여부

인간 평가 지표는 5점 척도로 기록한다.

* 코드 스멜 제거 정도: 대상 코드 스멜이 실제로 완화되었는가?
* 행동 보존 신뢰도: 테스트 외에도 원래 의도와 의미가 유지되는 것으로 보이는가?
* 가독성: 함수명, 변수명, 구조가 이해하기 쉬운가?
* 설계 적절성: 책임 분리와 추상화 수준이 과도하거나 부족하지 않은가?
* 변경 최소성: 리팩토링 목적과 무관한 기능 추가 또는 삭제가 없는가?

최종 성공 기준은 "테스트를 모두 통과하고, 인간 평가에서 코드 스멜 제거 정도와 가독성이 각각 4점 이상인 경우"로 정의한다. 이 기준을 만족하는 비율을 모델별 리팩토링 성공률로 계산한다.

3.1.7 결과 분석 방법
결과 분석은 모델별, 코드 스멜 유형별, 프롬프트 조건별로 수행한다. 먼저 테스트 통과율과 실행 가능률을 통해 기능 안정성을 비교하고, 그 다음 복잡도와 중복도 개선 정도를 통해 정량적 품질 개선 여부를 확인한다. 마지막으로 인간 평가 결과를 사용하여 자동 지표만으로는 확인하기 어려운 설계 품질을 분석한다.

특히 다음과 같은 실패 유형을 별도로 분류한다.

* 기능 변경: 테스트 실패 또는 입력/출력 의미 변화
* 과도한 리팩토링: 요구하지 않은 아키텍처 변경, 불필요한 클래스 추가
* 불충분한 리팩토링: 이름만 바꾸고 코드 스멜은 남아 있는 경우
* 환각: 존재하지 않는 라이브러리, 함수, 요구사항을 생성한 경우
* API 파괴: 기존 함수명, 인자, 반환 형식을 변경한 경우

이 실패 유형 분석은 단순한 점수 비교보다 LLM 기반 자동 리팩토링의 실제 한계를 설명하는 데 중요하다.

3.1.8 실험 산출물
최종 보고서에는 다음 산출물을 포함한다.

* 실험 데이터셋 요약표
* 모델별 테스트 통과율 및 실행 가능률 표
* 코드 스멜 유형별 리팩토링 성공률 표
* 프롬프트 방식별 성능 비교표
* 대표 성공 사례 1~2개
* 대표 실패 사례 1~2개
* LLM 리팩토링의 실무 적용 가능성과 한계에 대한 논의
