# LLM 자동 리팩토링 실험 계획

## 1. 실험 목적

LLM이 코드 스멜이 있는 Java 코드를 리팩토링했을 때, 원래 동작을 유지하면서 코드 구조를 개선할 수 있는지 확인한다. 본 실험은 복잡한 도구 체계보다 다음 두 질문에 집중한다.

- 리팩토링 후에도 기존 테스트가 통과하는가?
- 코드 스멜을 줄이는 방향으로 구조가 단순해졌는가?

## 2. 실험 대상

실험 대상은 `llm-refactoring-experiment` 프로젝트의 세 가지 코드 스멜 샘플이다.

| 코드 스멜 | 원본 코드 위치 | 검증 테스트 위치 |
| --- | --- | --- |
| Duplicated Code | `src/main/java/experiment/duplicatedcode/` | `src/test/java/experiment/duplicatedcode/` |
| Large Class | `src/main/java/experiment/largeclass/` | `src/test/java/experiment/largeclass/` |
| Long Method | `src/main/java/experiment/longmethod/` | `src/test/java/experiment/longmethod/` |

각 디렉터리에는 `DuplicateCode01`부터 `DuplicateCode50`, `LargeClass01`부터 `LargeClass50`, `LongMethod01`부터 `LongMethod50`까지 총 150개의 샘플이 있다. 시간 제약을 고려하여 1차 실험은 각 코드 스멜별 10개, 총 30개 샘플로 진행한다. 필요하면 최종 보고서에서 각 코드 스멜별 20개 또는 50개로 확장한다.

## 3. 테스트 작성 기준

LLM 리팩토링 결과를 정량적으로 비교하려면 먼저 원본 코드의 동작을 고정해야 한다. 각 샘플마다 JUnit 5 테스트를 작성하고, 리팩토링 전 원본 코드에서 모두 통과하는지 확인한다.

테스트는 다음 기준으로 작성한다.

- public 메서드의 대표 입력과 경계 입력을 검증한다.
- 반환값뿐 아니라 객체 상태 변화도 검증한다.
- Duplicated Code 샘플은 중복된 메서드들이 같은 결과를 내는지 확인한다.
- Large Class 샘플은 주요 상태 변경 메서드 호출 후 `summary`, `tier`, `notes` 등 관찰 가능한 결과를 확인한다.
- Long Method 샘플은 조건 분기를 통과하는 여러 입력을 사용하여 출력 문자열 또는 계산 결과를 확인한다.

테스트가 실패하는 리팩토링 결과는 기능 보존에 실패한 것으로 판정한다.

## 4. LLM 리팩토링 절차

각 샘플에 대해 동일한 절차를 반복한다.

1. 원본 샘플과 해당 테스트를 준비한다.
2. `mvn test`로 원본 코드가 테스트를 통과하는지 확인한다.
3. LLM에게 원본 Java 파일을 제공하고 리팩토링을 요청한다.
4. LLM 결과를 같은 클래스명과 public API가 유지되도록 적용한다.
5. 다시 `mvn test`를 실행하여 동작 보존 여부를 확인한다.
6. 통과한 결과에 대해서만 코드 구조 개선 정도를 기록한다.

LLM에는 다음 제약을 명시한다.

- public 클래스명, public 메서드명, 파라미터, 반환 타입을 바꾸지 않는다.
- 테스트 코드가 기대하는 외부 동작을 바꾸지 않는다.
- 새로운 외부 라이브러리를 추가하지 않는다.
- 코드 스멜 제거에 필요한 최소한의 구조 변경만 수행한다.

## 5. 평가 지표

평가는 기능 보존 여부를 1차 기준으로 삼고, 통과한 결과에 대해서만 구조 개선을 비교한다.

| 지표 | 측정 방법 | 판정 의미 |
| --- | --- | --- |
| 테스트 통과 여부 | `mvn test` | 리팩토링 후 동작 보존 여부 |
| 컴파일 성공 여부 | Maven 컴파일 | LLM 결과가 실행 가능한 Java 코드인지 여부 |
| 메서드 수 | 리팩토링 전후 public/private 메서드 수 비교 | Extract Method 등 구조 분해 여부 |
| 최대 메서드 길이 | 가장 긴 메서드의 라인 수 비교 | Long Method 개선 여부 |
| 중복 코드 감소 여부 | 유사 로직이 공통 메서드로 분리되었는지 확인 | Duplicated Code 개선 여부 |
| 클래스 책임 분리 여부 | 필드와 메서드가 보조 클래스로 분리되었는지 확인 | Large Class 개선 여부 |

최종 성공률은 다음과 같이 계산한다.

```text
기능 보존 성공률 = 테스트 통과 샘플 수 / 전체 리팩토링 샘플 수
구조 개선 성공률 = 테스트를 통과하고 코드 스멜도 완화된 샘플 수 / 전체 리팩토링 샘플 수
```

## 6. 결과 기록

각 샘플마다 다음 항목만 표로 기록한다.

| sample_id | smell_type | test_before | test_after | refactoring_result | note |
| --- | --- | --- | --- | --- | --- |
| DC01 | Duplicated Code | pass | pass/fail | improved/unchanged/broken | 간단한 관찰 |

`refactoring_result`는 다음 세 값으로 분류한다.

- `improved`: 테스트를 통과하고 코드 스멜이 줄어든 경우
- `unchanged`: 테스트는 통과했지만 구조 개선이 거의 없는 경우
- `broken`: 컴파일 실패 또는 테스트 실패가 발생한 경우

## 7. 최종 분석 방향

최종 보고서에서는 코드 스멜별로 다음 내용을 비교한다.

- 어떤 코드 스멜에서 LLM 리팩토링이 가장 안정적인가?
- 실패한 경우는 컴파일 오류, API 변경, 동작 변경 중 어느 유형이 많은가?
- 테스트가 LLM 리팩토링의 오류를 얼마나 잘 잡아냈는가?
- 자동 리팩토링을 실제 개발에 적용하려면 어떤 제약 조건이 필요한가?

이 실험은 LLM의 리팩토링 능력을 과도하게 일반화하지 않고, 테스트로 검증 가능한 작은 Java 샘플에서 기능 보존과 구조 개선 가능성을 확인하는 데 초점을 둔다.
