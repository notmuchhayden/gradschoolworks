# AI 기반 코드 스멜(Code Smell) 탐지 및 자동 리팩토링 성능 분석

## 초록

최근 대규모 언어모델(Large Language Model, LLM)의 발전으로 소스 코드 생성, 테스트 작성, 오류 수정 등 다양한 소프트웨어공학 작업이 자동화되고 있다. 그러나 기존 코드를 기능적으로 동일하게 유지하면서 내부 구조를 개선하는 리팩토링 작업은 단순 코드 생성보다 더 높은 안정성과 맥락 이해 능력을 요구한다. 특히 LLM 은 자연어 설명과 코드 패턴을 바탕으로 유연한 수정을 제안할 수 있지만, 기능 변경, 과도한 구조 변경, 존재하지 않는 API 생성, 테스트 미통과와 같은 위험도 함께 가진다.

본 연구는 LLM 이 대표적인 코드 스멜(Code Smell)을 포함한 코드를 얼마나 효과적으로 탐지하고 리팩토링할 수 있는지를 분석하기 위한 실험 설계를 제안한다. 실험 대상 코드 스멜은 긴 메서드(Long Method), 중복 코드(Duplicated Code), 큰 클래스 또는 데이터 클래스(Large Class/Data Class)로 한정한다. Java 기반 샘플 코드와 JUnit 테스트를 준비하고, 여러 LLM 에 동일한 리팩토링 요청을 수행한 뒤 컴파일 가능률, 테스트 통과율, 복잡도 변화, 코드 스멜 감소 여부, 인간 평가 루브릭을 통해 결과를 비교한다. 이를 통해 LLM 기반 자동 리팩토링의 가능성과 한계를 소프트웨어공학 관점에서 평가하고자 한다.

**주요어:** 대규모 언어모델, 코드 스멜, 자동 리팩토링, 소프트웨어 유지보수, 정적분석

## 1. 서론

### 1.1 연구 배경

리팩토링은 소프트웨어의 외부 동작은 유지하면서 내부 구조를 개선하는 활동이다. 기능 추가와 오류 수정이 지속되는 소프트웨어 프로젝트에서 리팩토링은 유지보수성, 가독성, 확장성을 확보하기 위한 핵심 활동이다. 그러나 실제 개발 현장에서는 일정 압박, 테스트 부족, 레거시 코드의 복잡성 때문에 리팩토링이 충분히 수행되지 못하는 경우가 많다.

최근 LLM 은 코드 생성과 설명 능력에서 빠르게 발전하고 있으며, 개발자는 ChatGPT, Claude, Gemini, GitHub Copilot 등 다양한 도구를 통해 코드를 작성하거나 수정하고 있다. 이러한 흐름은 리팩토링 작업에도 적용될 수 있다. LLM 은 긴 메서드를 여러 함수로 나누거나, 중복 로직을 공통 함수로 추출하거나, 복잡한 조건문을 읽기 쉽게 정리하는 등의 작업을 자연어 요청만으로 수행할 수 있다.

하지만 리팩토링은 기능 구현과 달리 기존 동작 보존이 핵심 조건이다. LLM 이 코드 구조를 보기 좋게 바꾸더라도 테스트를 통과하지 못하거나, 공개 API 를 바꾸거나, 요구하지 않은 기능을 추가한다면 실무적으로 사용할 수 없다. 따라서 LLM 기반 리팩토링의 성능은 단순히 결과 코드가 그럴듯한지가 아니라, 기능 보존과 품질 개선이 함께 달성되는지를 기준으로 평가해야 한다.

### 1.2 연구 목적

본 연구의 목적은 LLM 이 코드 스멜을 포함한 코드를 입력받았을 때 리팩토링 작업을 어느 정도 신뢰성 있게 수행할 수 있는지 확인하는 것이다. 구체적으로 다음 세 가지를 확인한다.

첫째, LLM 이 대표적인 코드 스멜에 대해 적절한 리팩토링 방향을 제안하는지 확인한다. 둘째, LLM 이 생성한 리팩토링 결과가 컴파일되고 기존 테스트를 통과하여 기능적 동등성을 유지하는지 확인한다. 셋째, 리팩토링 이후 코드 복잡도, 중복도, 유지보수성 지표가 실제로 개선되는지 분석한다.

## 2. 관련 연구

### 2.1 LLM 기반 자동 리팩토링

Kim 과 Li 의 연구인 “EM-Assist: Safe Automated Extract Method Refactoring with LLMs”는 LLM 을 활용하여 Extract Method 리팩토링을 지원하는 도구를 제안한다. 이 연구는 LLM 의 생성 능력과 정적분석 기반 검증을 결합하여 여러 리팩토링 후보를 생성하고 순위화한다. 오픈소스 프로젝트에서 수집한 리팩토링 사례를 대상으로 평가한 결과, LLM 기반 접근이 기존 정적분석 중심 도구보다 더 높은 추천 성능을 보일 수 있음을 보고하였다.

이 연구의 의의는 LLM 을 단순 코드 생성기가 아니라 실제 IDE 환경에 통합 가능한 리팩토링 보조 도구로 다루었다는 점이다. 동시에 리팩토링 제안을 안전하게 사용하기 위해서는 정적분석과 테스트 기반 검증이 함께 필요하다는 점을 보여준다.

### 2.2 LLM 기반 리팩토링의 한계

Cordeiro, Noei, Zou 의 “LLM-Driven Code Refactoring: Opportunities and Limitations”는 LLM 기반 리팩토링의 가능성과 한계를 정리한 연구이다. 이 연구는 LLM 이 코드 구조를 개선할 잠재력을 가지지만, 맥락 이해 부족, 환각, 목적 불일치, 테스트 부족 등의 문제가 존재한다고 지적한다.

특히 리팩토링은 코드의 의미를 보존해야 하는 작업이므로, LLM 의 출력이 문법적으로 올바르더라도 실질적으로는 잘못된 결과일 수 있다. 예를 들어 함수명을 바꾸거나 인자 구조를 변경하면 내부 코드는 더 간결해질 수 있지만 기존 호출부와의 호환성이 깨질 수 있다. 따라서 LLM 기반 리팩토링의 평가는 코드 품질 지표뿐 아니라 기능 보존 여부를 반드시 포함해야 한다.

### 2.3 테스트 주도 리팩토링과 하네스 엔지니어링

Smolić 등은 “AI-Assisted Unit Test Writing and Test-Driven Code Refactoring: A Case Study”에서 AI 를 활용한 테스트 작성과 리팩토링 사례를 제시한다. 이 연구에서는 LLM 을 단독으로 사용하는 것이 아니라, 계획자와 수행자를 분리하는 에이전트 방식 및 명시적 제약을 제공하는 하네스 엔지니어링을 활용한다.

하네스 엔지니어링은 LLM 에게 작업 규칙, 금지 사항, 출력 형식, 테스트 조건 등을 명시하여 예측 가능한 결과를 유도하는 방식이다. 본 연구에서도 이 관점을 반영하여 기본 프롬프트와 제약 프롬프트를 비교한다. 이를 통해 모델 자체 성능뿐 아니라 프롬프트 제약이 리팩토링 안정성에 미치는 영향을 함께 분석한다.

본 연구는 다음 네 가지 연구 문제를 중심으로 수행한다.

| 연구 문제 | 내용 |
| --- | --- |
| RQ1 | LLM 은 대표적인 코드 스멜을 포함한 코드에서 적절한 리팩토링 방향을 제안할 수 있는가? |
| RQ2 | LLM 이 생성한 리팩토링 결과는 기존 기능을 보존하는가? |

| RQ4 | 모델 종류와 프롬프트 방식에 따라 리팩토링 품질에는 어떤 차이가 발생하는가? |

## 4. 연구 방법

### 4.1 실험 대상 코드 스멜

모든 코드 스멜을 대상으로 실험하면 데이터셋 구축과 인간 평가 부담이 지나치게 커진다. 따라서 본 연구에서는 다음 세 가지 코드 스멜을 우선 실험 대상으로 선정한다.

| 코드 스멜 | 설명 | 주요 리팩토링 기법 |
| --- | --- | --- |
| 긴 메서드(Long Method) | 하나의 함수나 메서드가 여러 책임을 동시에 수행하는 경우 | Extract Method, 조건문 분리, 변수명 개선 |
| 중복 코드(Duplicated Code) | 유사한 로직이 여러 위치에 반복되는 경우 | 공통 함수 추출, 파라미터화, 템플릿 메서드 |
| 큰 클래스/데이터 클래스(Large Class/Data Class) | 하나의 클래스가 과도한 책임을 갖거나 데이터만 보유하는 경우 | Extract Class, 책임 분리, 캡슐화 |

선정 기준은 대표성, 측정 가능성, 실험 가능성이다. 긴 메서드와 중복 코드는 리팩토링 문헌에서 가장 자주 언급되는 코드 스멜이며, 복잡도와 중복도 지표로 변화량을 측정하기 쉽다. 큰 클래스 또는 데이터 클래스는 객체지향 설계 관점에서 중요한 코드 스멜이므로 함께 포함한다.

### 4.2 실험 언어와 도구

실험 언어는 Java 로 통일한다. Java 는 정적 타입 언어이며 클래스, 인터페이스, 접근 제어자, 패키지 구조가 명확하므로 코드 스멜과 객체지향 리팩토링을 표현하기에 적합하다. 또한 JUnit, Maven 또는 Gradle, SonarQube, PMD, Checkstyle, SpotBugs 등 검증 도구가 풍부하여 LLM 이 생성한 리팩토링 결과를 컴파일, 테스트, 정적분석 관점에서 체계적으로 평가할 수 있다.

Java 를 선택한 또 다른 이유는 기존 코드 스멜 및 리팩토링 연구에서 Java 가 널리 사용되어 왔다는 점이다. 긴 메서드, 큰 클래스, 데이터 클래스, 중복 코드와 같은 대표적 코드 스멜은 Java 의 객체지향 구조에서 명확하게 드러나며, Extract Method, Extract Class, Encapsulate Field 와 같은 리팩토링 기법도 Java 예제로 설명하기 쉽다.

| 구분 | 사용 도구 | 목적 |
| --- | --- | --- |
| 빌드/테스트 | Maven 또는 Gradle, JUnit 5 | 리팩토링 전후 컴파일 및 기능 동등성 검증 |
| 정적분석 | SonarQube 또는 SonarLint | 코드 스멜, 복잡도, 중복도, 유지보수성 관련 지표 측정 |
| 코드 품질 검사 | PMD, Checkstyle, SpotBugs | 규칙 위반, 잠재 버그, 스타일 문제 확인 |
| 결과 정리 | CSV 또는 스프레드시트 | 모델별 결과 기록 및 분석 |

### 4.3 데이터셋 구성

데이터셋은 코드 스멜 유형별 10개, 총 30개 샘플을 목표로 구성한다. 시간 제약이 있을 경우 최소 범위는 코드 스멜 2종, 총 20개 샘플로 축소한다.

각 샘플은 다음 파일과 메타데이터를 포함한다.

| 항목 | 설명 |
| --- | --- |
| 원본 코드 | 코드 스멜을 포함한 Java 코드 |
| 테스트 코드 | 원본 코드의 기대 동작을 검증하는 JUnit 테스트 |
| 코드 스멜 라벨 | Long Method, Duplicated Code, Large Class/Data Class 중 하나 |
| 난이도 | easy, medium, hard 중 하나 |
| 평가 메모 | 사람이 판단한 주요 문제점과 기대 리팩토링 방향 |

샘플은 공개 예제, 리팩토링 교재의 예시를 변형한 코드, 직접 작성한 소형 프로그램을 중심으로 구성한다. LLM 으로 생성한 코드는 데이터가 부족할 때 보조적으로 사용하되, 실험에 포함하기 전 사람이 코드 스멜 존재 여부와 테스트 타당성을 검토한다.

### 4.4 비교 대상 모델

비교 대상은 실험 시점에 접근 가능한 상용 LLM 2~3개로 구성한다. 기본적으로 ChatGPT 계열, Claude 계열, Gemini 계열 중 하나씩 선택한다. 가능하다면 오픈소스 또는 로컬 코딩 모델 1개를 추가하여 상용 모델과 비교한다.

| 모델 구분 | 역할 |
| --- | --- |
| ChatGPT 계열 | 범용 코딩 및 설명 능력을 가진 기준 모델 |
| Claude 계열 | 긴 문맥 처리와 코드 설명 능력을 가진 비교 모델 |
| Gemini 계열 | 다른 제공사의 상용 모델 비교 대상 |
| 로컬/오픈소스 모델 | 비용과 개인정보 측면에서 대안이 될 수 있는 확장 실험 대상 |

모든 모델에는 동일한 코드 샘플과 동일한 프롬프트를 제공한다. 모델별 응답 횟수도 동일하게 맞추어 비교 가능성을 확보한다.

### 4.5 프롬프트 조건

본 연구는 프롬프트 방식이 리팩토링 결과에 미치는 영향을 확인하기 위해 두 가지 조건을 비교한다.

| 조건 | 설명 |
| --- | --- |
| 기본 프롬프트 | 코드와 리팩토링 요청만 제공한다. |
| 제약 프롬프트 | 테스트 통과, 공개 API 유지, 불필요한 기능 추가 금지, 변경 요약 작성 등 명시적 제약을 포함한다. |

기본 프롬프트 예시는 다음과 같다.

```text
다음 Java 코드의 기능을 유지하면서 코드 스멜을 제거하도록 리팩토링하라.
리팩토링된 코드만 제시하라.
```

제약 프롬프트 예시는 다음과 같다.

```text
다음 Java 코드를 리팩토링하라.
조건은 다음과 같다.
1. 기존 public 클래스명, public 메서드명, 인자, 반환 형식을 변경하지 말 것.
2. 기존 테스트가 모두 통과해야 한다.
3. 새로운 외부 라이브러리나 빌드 설정 변경을 추가하지 말 것.
4. 기능 추가나 삭제를 하지 말 것.
5. 코드 스멜을 줄이되 과도한 구조 변경은 피할 것.
6. 마지막에 변경 요약을 3줄 이내로 작성할 것.
```

## 5. 실험 절차

각 샘플에 대해 다음 절차를 동일하게 적용한다.

1. 원본 코드와 테스트 케이스를 준비한다.
2. 원본 코드가 컴파일되고 모든 JUnit 테스트를 통과하는지 확인한다.
3. 원본 코드의 LOC, cyclomatic complexity, cognitive complexity, 중복도, 정적분석 위반 수를 측정한다.
4. 동일한 코드와 동일한 프롬프트를 각 LLM 에 입력한다.
5. LLM 이 생성한 결과를 모델명과 샘플 번호가 포함된 별도 디렉터리에 저장한다.
6. 마크다운 코드 블록 제거, 파일명 정리 등 최소한의 형식 정리만 수행한다.
7. 리팩토링 결과에 대해 컴파일 검사와 JUnit 테스트를 실행한다.
8. 리팩토링 결과의 정적 지표를 다시 측정한다.
9. 인간 평가자가 루브릭에 따라 결과를 평가한다.
10. 실패 사례는 실패 유형별로 분류한다.

실험 과정에서 사람이 LLM 의 결과 코드를 직접 수정하면 모델 성능 평가가 왜곡될 수 있다. 따라서 패키지 선언 정리나 코드 블록 제거처럼 실행을 위한 최소한의 형식 수정만 허용하고, 로직 변경은 수행하지 않는다.

## 6. 평가 지표

### 6.1 자동 평가 지표

| 지표 | 설명 |
| --- | --- |
| 컴파일 가능률 | 리팩토링 결과가 Java 컴파일을 통과하는 비율 |
| 테스트 통과율 | 리팩토링 후 기존 JUnit 테스트를 모두 통과한 비율 |
| 복잡도 변화율 | 리팩토링 전후 cyclomatic complexity 또는 cognitive complexity 변화 |
| 코드 스멜 변화 | SonarQube 또는 PMD 가 탐지한 코드 스멜/규칙 위반 수 변화 |
| 코드 크기 변화 | LOC 증가 또는 감소량 |
| 중복도 변화 | 반복 코드 블록의 감소 여부 |

### 6.2 인간 평가 지표

인간 평가는 5점 척도로 수행한다. 1점은 매우 낮음, 5점은 매우 높음을 의미한다.

| 평가 항목 | 평가 기준 |
| --- | --- |
| 코드 스멜 제거 정도 | 대상 코드 스멜이 실제로 완화되었는가? |
| 행동 보존 신뢰도 | 테스트 외에도 원래 의미가 유지되는 것으로 보이는가? |
| 가독성 | 함수명, 변수명, 구조가 이해하기 쉬운가? |
| 설계 적절성 | 책임 분리와 추상화 수준이 적절한가? |
| 변경 최소성 | 리팩토링 목적과 무관한 변경이 없는가? |

최종 성공 기준은 다음과 같이 정의한다.

```text
성공 = Java 컴파일을 통과하고,
       기존 JUnit 테스트를 모두 통과하고,
       코드 스멜 제거 정도가 4점 이상이며,
       가독성이 4점 이상인 경우
```

이 기준을 만족하는 샘플 비율을 모델별 리팩토링 성공률로 계산한다.

## 8. 예상되는 분석 관점

### 8.1 기능 보존과 코드 품질 개선의 긴장 관계

LLM 리팩토링 결과는 외형상 더 깔끔해 보이더라도 컴파일 또는 테스트를 통과하지 못할 수 있다. 반대로 테스트는 통과하지만 코드 스멜이 거의 제거되지 않는 경우도 가능하다. 따라서 본 연구에서는 컴파일 가능성, 기능 보존, 품질 개선을 별도로 측정하고, 이를 동시에 만족하는 경우만 성공으로 판단한다.

### 8.2 코드 스멜 유형별 난이도 차이

긴 메서드는 함수 추출과 조건문 분리를 통해 비교적 명확하게 개선될 수 있다. 중복 코드는 반복 패턴을 정확히 인식해야 하므로 중복의 형태가 조금씩 다를수록 난이도가 올라간다. 큰 클래스 또는 데이터 클래스는 책임 분리와 설계 판단이 필요하므로 단순 코드 변환보다 더 어렵다. 따라서 코드 스멜 유형에 따라 LLM 성능이 다르게 나타날 가능성이 높다.

### 8.3 프롬프트 제약의 효과

제약 프롬프트는 LLM 이 공개 API 를 변경하거나 불필요한 기능을 추가하는 일을 줄일 수 있다. 그러나 제약이 지나치게 강하면 모델이 보수적으로 동작하여 충분한 리팩토링을 수행하지 못할 수도 있다. 따라서 본 연구에서는 기본 프롬프트와 제약 프롬프트의 차이를 비교하여 하네스 엔지니어링의 효과를 분석한다.

### 8.4 실패 유형 분석

실패 사례는 다음 유형으로 분류한다.

| 실패 유형 | 설명 |
| --- | --- |
| 기능 변경 | JUnit 테스트 실패 또는 입력/출력 의미 변화 |
| 과도한 리팩토링 | 요구하지 않은 아키텍처 변경, 불필요한 클래스 추가 |
| 불충분한 리팩토링 | 이름만 바꾸고 코드 스멜은 남아 있는 경우 |
| 환각 | 존재하지 않는 라이브러리, 클래스, 메서드, 요구사항을 생성한 경우 |
| API 파괴 | 기존 public 클래스명, 메서드명, 인자, 반환 형식을 변경한 경우 |

이 분류는 단순한 평균 점수보다 실무 적용 가능성을 설명하는 데 더 유용하다.

## 9. 연구의 한계

본 연구는 제한된 기간의 텀프로젝트로 수행되므로 몇 가지 한계를 가진다.

첫째, 데이터셋 규모가 크지 않다. 코드 스멜 유형별 10개 내외의 샘플은 모델의 전반적 성능을 일반화하기에는 부족할 수 있다. 둘째, Java 코드만 대상으로 하기 때문에 Python, C++, JavaScript 등 다른 언어의 리팩토링 성능과 차이가 있을 수 있다. 셋째, 인간 평가가 포함되므로 평가자의 주관이 결과에 영향을 줄 수 있다. 이를 완화하기 위해 명시적 루브릭을 사용하지만, 완전히 객관적인 평가는 어렵다. 넷째, 상용 LLM 은 실험 시점의 모델 버전에 따라 결과가 달라질 수 있다.

이러한 한계에도 불구하고 본 연구는 LLM 기반 자동 리팩토링을 기능 보존, 정적 지표, 인간 평가를 함께 사용하여 분석한다는 점에서 의미가 있다.

## 10. 결론

본 연구는 LLM 이 코드 스멜 탐지 및 자동 리팩토링 작업에서 어느 정도 활용 가능한지를 평가하기 위한 실험 설계를 제시하였다. 리팩토링은 단순 코드 생성과 달리 기존 동작 보존이 필수적이므로, 컴파일 가능 여부, 테스트 통과 여부, 코드 품질 개선 여부를 함께 측정해야 한다. 이를 위해 본 연구는 Java 기반 샘플 코드와 JUnit 테스트를 구성하고, 긴 메서드, 중복 코드, 큰 클래스 또는 데이터 클래스에 대해 여러 LLM 의 리팩토링 결과를 비교하는 방식을 제안하였다.

예상되는 핵심 쟁점은 LLM 이 코드 구조를 개선하는 능력을 보이더라도 항상 기능을 보존하지는 못한다는 점이다. 따라서 LLM 기반 리팩토링은 독립적인 자동화 도구라기보다 테스트, 정적분석, 인간 검토와 결합된 보조 도구로 활용하는 것이 현실적이다. 최종 실험을 통해 모델별 강점과 약점, 코드 스멜 유형별 난이도, 프롬프트 제약의 효과를 분석하면 LLM 리팩토링의 실무 적용 가능성과 한계를 보다 구체적으로 제시할 수 있을 것이다.
