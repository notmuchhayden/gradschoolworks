# LLM 기반 코드 리팩토링 성능 검증: 로컬 및 경량 LLM 모델 비교

## 요약
본 연구는 LLM을 활용한 자동 코드 리팩토링이 기존 동작을 유지하면서 코드 스멜을 완화할 수 있는지 실험적으로 검증한다. 중간보고서 단계에서는 최근 AI 기반 코드 리팩토링 연구를 검토하고, 자동 리팩토링의 가능성과 한계를 조사하였다. 최종 실험에서는 연구 범위를 로컬 또는 경량 LLM의 리팩토링 성능 검증으로 구체화하였다. 실험 대상은 Java 코드의 대표적인 코드 스멜인 중복 코드, 긴 메서드, 거대 클래스 세 유형이며, 각 유형별 10개씩 총 30개 샘플을 사용하였다. 각 샘플에 대해 qwen2.5-coder-7b-instruct-q6_k, nvidia/nemotron-3-nano-30b-a3b, google/gemma-4-31b-it 세 모델로 리팩토링을 수행하고, 총 90개 결과를 테스트하였다.

평가는 리팩토링 후 코드가 원본과 같은 동작을 유지하는지에 초점을 두었다. 컴파일 실패, 클래스 또는 메서드 누락, 런타임 예외, 원본과 다른 출력은 모두 실패로 계산하였다. 실험 결과 전체 성공률은 64.4%였으며, 모델별 성공률은 gemma4 76.7%, qwen25 63.3%, namotron3 53.3%였다. 코드 유형별로는 중복 코드 86.7%, 긴 메서드 80.0%, 거대 클래스 26.7%의 성공률을 보였다. 이 결과는 LLM이 국소적인 리팩토링에는 비교적 유용하지만, 복잡한 상태와 책임 분리가 필요한 거대 클래스 리팩토링에는 아직 한계가 크다는 점을 보여준다.

## 1. 서론
리팩토링은 소프트웨어의 외부 동작을 유지하면서 내부 구조를 개선하는 활동이다. 코드 중복 제거, 긴 메서드 분해, 거대 클래스 분리와 같은 리팩토링은 유지보수성, 가독성, 확장성을 높이는 데 중요하다. 그러나 실제 개발 현장에서 리팩토링은 많은 시간과 주의를 요구한다. 특히 기존 동작을 보존해야 하므로 단순히 코드를 새로 작성하는 것보다 더 엄격한 검증이 필요하다.

최근 LLM은 코드 생성, 테스트 작성, 버그 수정, 리팩토링 제안 등 다양한 소프트웨어공학 작업에 활용되고 있다. LLM은 자연어 지시를 바탕으로 코드를 수정할 수 있기 때문에 기존 정적 분석 기반 리팩토링 도구보다 유연해 보인다. 반면 LLM은 환각을 일으키거나, 원본 코드의 공개 API를 바꾸거나, 컴파일되지 않는 코드를 생성할 수 있다. 따라서 LLM 기반 리팩토링의 실용성을 판단하려면 단순한 사례 제시가 아니라 테스트 기반의 검증이 필요하다.

본 연구는 다음 질문에 답하고자 한다.

1. LLM은 리팩토링 후에도 원본 코드의 동작을 유지할 수 있는가?
2. 코드 스멜 유형에 따라 LLM 리팩토링 성공률은 어떻게 달라지는가?
3. 로컬 또는 경량 LLM을 자동 리팩토링에 활용할 때 어떤 한계가 나타나는가?

## 2. 관련 연구
### 2.1 자동 리팩토링의 효과
참고 문헌의 논문 “EM-Assist: Safe Automated Extract Method Refactoring with LLMs”[1]은 실제로 동작하는 제품 수준의 자동 리팩토링 툴을 제작하면서 LLM을 사용한 리팩토링 툴의 유용성을 검증한다.

EM-Assist는 LLM을 통하여 긴 메서드에 Extract Method를 수행하여 짧은 여러 개의 코드로 분할하는 툴이다. 상용 코드 에디터인 IntelliJ 플러그인 형태로 제공된다.

특징적인 부분은 LLM을 통한 리팩토링에 더해 정적 분석을 동시에 수행하여, 여러 개의 제안을 만들어서 순위화하는 메커니즘을 제공한다는 점이다. 오픈소스에서 실제 수행된 1,752건의 리팩토링 결과, 상위 5개의 제안 내에서 53.4%의 비율로 제대로 된 리팩토링 제안을 하였다고 한다. 이 결과는 기존의 정적 분석 도구의 39.4% 대비 큰 개선이다. 또한 개발자 설문에서 94.4%의 긍정적 평가를 얻었다고 주장한다.

이 연구는 LLM 리팩토링이 단순한 코드 생성이 아니라 정적 분석, 후보 생성, 순위화와 결합될 때 더 안전하게 사용될 수 있음을 보여준다. 본 연구의 실험도 리팩토링 결과를 테스트로 검증해야 한다는 관점에서 이 연구와 연결된다.

### 2.2 LLM 기반 코드 리팩토링의 문제점
참고 문헌의 “LLM-Driven Code Refactoring: Opportunities and Limitations”[2] 논문은 LLM에 기반한 코드 리팩토링의 가능성과 한계를 분석한 포지션 페이퍼이다. 새로운 실험 결과를 제시하기보다는 LLM을 통한 리팩토링을 수행할 때 예상될 수 있는 여러 문제점을 지적한다.

첫 번째 문제는 IDE 통합의 어려움과 제한된 맥락 인식 능력이다. 기존 IDE는 이미 구문 강조, 코드 제안, 리팩토링 기능을 포함하는 경우가 많다. 여기에 LLM 기반 리팩토링 기능을 통합하면 LLM의 느린 속도로 인한 반응성 문제가 발생할 수 있다. 또한 리팩토링 작업은 코드의 맥락을 인식하며 안전하게 변경하는 것이 목표인데, LLM의 맥락 인식 능력이 부족하면 잘못된 리팩토링을 수행할 가능성이 있다.

두 번째 문제는 환각 처리의 어려움이다. 기존 IDE의 리팩토링 기능은 제한된 규칙 안에서만 동작하지만, LLM은 규칙에서 벗어난 코드를 생성할 수 있다. 이 경우 예상하지 못한 동작 변화가 발생할 수 있다.

세 번째 문제는 리팩토링 목적에 대한 이해 부족이다. 인간 개발자는 코드 스멜 제거, 가독성 향상, 설계 원칙 준수 등 특정 목적을 달성하기 위해 코드를 리팩토링한다. 그러나 LLM은 수정에 대한 명시적인 근거 없이 동작하기 때문에 프로젝트의 목표와 맞지 않는 변경을 제안할 수 있다.

네 번째 문제는 검증 테스트의 부족이다. LLM은 환각 현상을 겪을 수 있기 때문에 리팩토링 성공 여부를 검증하기 위한 테스트가 필수적이다. 테스트 케이스가 부족하면 LLM 리팩토링의 신뢰성을 판단하기 어렵고, 결국 인간 검증이 필요하다.

### 2.3 LLM 리팩토링의 실무 적용
참고 문헌 “AI-Assisted Unit Test Writing and Test-Driven Code Refactoring: A Case Study”[3]는 LLM을 통하여 자동으로 유닛 테스트를 생성하고, 이에 더해 자동 리팩토링까지 시도한 실제 상용 시스템 적용 사례 연구이다.

해당 논문에서는 Gemini 2.5 Pro와 여러 경량 모델을 사용하여 리팩토링을 진행하였다. 하나의 모델만 사용하는 것이 아니라 Agent 방식을 사용하여 여러 인공지능이 역할을 분담하도록 구성하였다. 특히 고성능 LLM은 계획자(Planner)로 사용하고 경량 모델은 수행자로 사용하여 임무별 역할을 분리하고 성능 부하를 줄였다.

또한 논문의 저자들은 유닛 테스트나 리팩토링 시 LLM에게 통일된 결과물을 유도하기 위해 하네스 엔지니어링을 시도하였다. 하네스 엔지니어링은 다양한 문서에 AI의 행동 제약 사항을 명시하여 AI가 예상하지 못한 임의의 행동을 하지 못하도록 제약하는 방법이다. 본 연구에서도 리팩토링 결과를 검증하기 위해 테스트 하네스를 구성했다는 점에서 이 접근과 유사한 문제의식을 가진다.

## 3. 연구 방법
### 3.1 연구 범위
중간보고서에서는 상용 LLM을 포함한 AI 리팩토링 전반을 실험 대상으로 고려하였다. 그러나 최종 실험에서는 로컬 또는 경량 모델의 리팩토링 성능 검증으로 범위를 좁혔다. 리팩토링은 많은 토큰을 소모하고 반복 실험이 필요하므로, 비용과 재현성을 고려할 때 로컬 또는 경량 모델 중심의 검증이 더 현실적이라고 판단하였다.

### 3.2 코드 스멜 선정
실험 대상 코드 스멜은 다음 세 가지이다.

| 코드 스멜 | 설명 | 주요 리팩토링 방향 |
| --- | --- | --- |
| 중복 코드 | 여러 위치에 유사한 로직이 반복되는 코드 | 공통 메서드 또는 공통 구조로 분리 |
| 긴 메서드 | 하나의 메서드에 많은 조건과 계산이 포함된 코드 | 의미 있는 작은 메서드로 분해 |
| 거대 클래스 | 하나의 클래스가 너무 많은 책임과 상태를 가진 코드 | 책임 분리, 응집도 향상, 결합도 완화 |

이 세 유형은 대표적인 코드 스멜이며, 리팩토링 난이도도 서로 다르다. 중복 코드와 긴 메서드는 비교적 국소적인 변환으로 해결 가능한 경우가 많지만, 거대 클래스는 상태 변화와 공개 API를 함께 보존해야 하므로 더 어렵다.

### 3.3 평가 기준
본 연구에서는 리팩토링 성공 여부를 기능 보존 관점에서 판단하였다. 리팩토링은 외부 동작을 바꾸지 않아야 하므로, 코드 구조가 개선된 것처럼 보이더라도 테스트를 통과하지 못하면 실패로 계산하였다.

실패 조건은 다음과 같다.

| 실패 조건 | 의미 |
| --- | --- |
| 컴파일 실패 | 리팩토링 결과 Java 소스가 컴파일되지 않음 |
| API 불일치 | 기대한 클래스 또는 공개 메서드가 없어 테스트에서 호출할 수 없음 |
| 동작 불일치 | 컴파일과 호출은 가능하지만 원본 샘플과 반환값이 다름 |
| 런타임 예외 | 원본은 처리한 입력에서 리팩토링 결과가 예외를 발생시킴 |

## 4. 실험 구성
### 4.1 실험 모델
실험에는 세 모델을 사용하였다.

| 모델 표기 | 모델명 | 실험 중 관찰 |
| --- | --- | --- |
| namotron3 | nvidia/nemotron-3-nano-30b-a3b | NVIDIA NIM 서비스 기반이며 매우 빠르지만 리팩토링 품질은 낮음 |
| qwen25 | qwen2.5-coder-7b-instruct-q6_k | Alibaba Cloud AI 서비스 기반이며 리팩토링 품질은 양호하지만 속도가 느림 |
| gemma4 | google/gemma-4-31b-it | Google AI 서비스 기반이며 리팩토링 품질과 속도 모두 양호 |

### 4.2 실험 절차
실험은 다음 순서로 진행하였다.

1. 중복 코드, 긴 메서드, 거대 클래스 샘플을 준비한다.
2. 각 코드 스멜별로 10개 샘플을 선택한다.
3. 세 모델에 동일한 유형의 리팩토링 요청을 수행한다.
4. 모델이 생성한 리팩토링 결과를 저장한다.
5. 테스트를 실행하여 원본과 리팩토링 결과의 행동 동등성을 검증한다.
6. 컴파일 실패, API 불일치, 동작 불일치, 런타임 예외를 실패로 기록한다.

### 4.3 프롬프트
중복 코드에는 기존 동작을 유지하면서 중복 로직을 제거하고, 공통 기능을 적절한 메서드나 구조로 분리하도록 요청하였다. 긴 메서드에는 기존 동작을 유지하면서 긴 메서드를 의미 있는 작은 메서드들로 분리하도록 요청하였다. 거대 클래스에는 하나의 클래스가 가진 여러 책임을 분석하고, 관련 기능을 적절한 클래스나 메서드로 분리하도록 요청하였다.

세 프롬프트 모두 기존 동작 유지, 가독성 향상, 유지보수성 개선을 공통 조건으로 포함하였다.

### 4.4 실험 규모
각 코드 유형은 모델별로 10개씩 테스트하였다. 따라서 모델별 테스트 수는 30개이고, 전체 테스트 수는 90개이다.

| 코드 유형 | 샘플 수 | 모델 수 | 테스트 수 |
| --- | ---: | ---: | ---: |
| 중복 코드 | 10 | 3 | 30 |
| 긴 메서드 | 10 | 3 | 30 |
| 거대 클래스 | 10 | 3 | 30 |
| 합계 | 30 | 3 | 90 |

## 5. 실험 결과
### 5.1 전체 결과
전체 90개의 리팩토링 결과 중 58개가 테스트를 통과했고, 32개가 실패하였다. 전체 성공률은 64.4%이다.

| 모델 | 테스트 수 | 성공 | 실패 | 성공률 |
| --- | ---: | ---: | ---: | ---: |
| qwen25 | 30 | 19 | 11 | 63.3% |
| namotron3 | 30 | 16 | 14 | 53.3% |
| gemma4 | 30 | 23 | 7 | 76.7% |
| 합계 | 90 | 58 | 32 | 64.4% |

모델별로는 gemma4가 가장 높은 성공률을 보였다. qwen25는 중간 수준의 성공률을 보였고, namotron3는 가장 낮은 성공률을 보였다. 특히 namotron3는 응답 속도는 빠르지만 복잡한 리팩토링에서 안정성이 부족하였다.

코드 유형별 합계는 다음과 같다.

| 코드 유형 | 테스트 수 | 성공 | 실패 | 성공률 |
| --- | ---: | ---: | ---: | ---: |
| 중복코드 | 30 | 26 | 4 | 86.7% |
| 긴 메서드 | 30 | 24 | 6 | 80.0% |
| 거대 클래스 | 30 | 8 | 22 | 26.7% |

중복 코드와 긴 메서드는 비교적 높은 성공률을 보였다. 이는 LLM이 반복 로직을 공통화하거나 긴 메서드를 작은 메서드로 나누는 국소적 리팩토링에는 강점을 가질 수 있음을 의미한다. 반면 거대 클래스는 성공률이 26.7%에 불과하였다. 거대 클래스 리팩토링은 단순히 클래스를 나누는 것이 아니라, 기존 공개 API와 상태 변화 규칙을 유지해야 하므로 LLM에게 더 어려운 작업이었다.

### 5.3 모델과 코드 유형별 결과
모델과 코드 유형을 함께 보면 다음과 같다.

| 코드 유형 | 모델 | 테스트 수 | 성공 | 실패 | 성공률 |
| --- | --- | ---: | ---: | ---: | ---: |
| 중복코드 | qwen25 | 10 | 7 | 3 | 70.0% |
| 중복코드 | namotron3 | 10 | 9 | 1 | 90.0% |
| 중복코드 | gemma4 | 10 | 10 | 0 | 100.0% |
| 긴 메서드 | qwen25 | 10 | 8 | 2 | 80.0% |
| 긴 메서드 | namotron3 | 10 | 7 | 3 | 70.0% |
| 긴 메서드 | gemma4 | 10 | 9 | 1 | 90.0% |
| 거대 클래스 | qwen25 | 10 | 4 | 6 | 40.0% |
| 거대 클래스 | namotron3 | 10 | 0 | 10 | 0.0% |
| 거대 클래스 | gemma4 | 10 | 4 | 6 | 40.0% |

gemma4는 중복 코드에서 100%, 긴 메서드에서 90%를 기록하여 가장 안정적인 결과를 보였다. qwen25는 중복 코드 70%, 긴 메서드 80%로 양호했지만 거대 클래스에서는 40%에 머물렀다. namotron3는 중복 코드에서는 90%로 높았으나, 거대 클래스에서는 0%로 모든 샘플이 실패하였다.

### 5.4 거대 클래스 리팩토링 분석
거대 클래스 리팩토링은 세 유형 중 가장 낮은 성공률을 보였다.

| 모델 | 성공 | 실패 | 성공률 | 판단 |
| --- | ---: | ---: | ---: | --- |
| qwen25 | 4 | 6 | 40.0% | 부분적으로만 적절함 |
| namotron3 | 0 | 10 | 0.0% | 부적절함 |
| gemma4 | 4 | 6 | 40.0% | 부분적으로만 적절함 |

qwen25는 LargeClass01, LargeClass02, LargeClass07, LargeClass08에서 테스트를 통과했지만, LargeClass03, LargeClass04, LargeClass10은 컴파일되지 않았고 LargeClass05, LargeClass06은 원본 공개 메서드가 유지되지 않았다. LargeClass09는 실행은 가능했지만 상태 변화 결과가 원본과 달랐다.

namotron3는 거대 클래스 10개 모두 실패하였다. 여러 결과에서 클래스명이 기대와 다르거나, 누락된 타입을 참조하거나, 컴파일되지 않는 코드가 생성되었다. 일부 코드는 실행까지 도달했더라도 원본과 다른 문자열 결과나 상태 값을 반환했다.

gemma4는 LargeClass01, LargeClass06, LargeClass07, LargeClass10에서 통과했지만, LargeClass02, LargeClass03, LargeClass04, LargeClass09는 기대한 클래스가 생성되지 않았고 LargeClass05, LargeClass08은 출력 형식이 원본과 달랐다.

종합하면 거대 클래스 리팩토링에서는 gemma4와 qwen25가 일부 샘플에서만 사용 가능한 결과를 냈고, namotron3는 신뢰하기 어려운 결과를 냈다.

## 6. 논의
### 6.1 LLM 리팩토링의 가능성
실험 결과 LLM은 중복 코드 제거와 긴 메서드 분해에서 비교적 높은 성공률을 보였다. 중복 코드의 전체 성공률은 86.7%, 긴 메서드의 전체 성공률은 80.0%였다. 이는 코드 변경 범위가 비교적 제한적이고, 리팩토링 목표가 명확한 경우 LLM이 실용적인 보조 도구가 될 수 있음을 보여준다.

특히 gemma4는 세 모델 중 가장 높은 전체 성공률을 보였으며, 중복 코드와 긴 메서드에서 안정적인 결과를 냈다. 따라서 로컬 또는 경량 LLM을 리팩토링에 활용할 때도 모델 선택에 따라 결과 품질 차이가 크게 나타난다.

### 6.2 LLM 리팩토링의 한계
거대 클래스 리팩토링은 전체 성공률이 26.7%에 불과하였다. 이 결과는 LLM이 단순한 코드 정리보다 구조적 설계 변경에 취약하다는 점을 보여준다. 거대 클래스 리팩토링에서는 클래스 책임 분리뿐 아니라 기존 public API 유지, 내부 상태 변화 보존, 출력 형식 유지가 모두 필요하다. 여러 실패 사례에서 LLM은 구조 분리를 시도했지만, 컴파일 오류를 만들거나 원본 메서드를 제거하거나 동작을 변경하였다.

따라서 LLM 리팩토링은 테스트 없이 자동 적용하기 어렵다. 특히 거대 클래스와 같이 영향 범위가 큰 리팩토링은 반드시 컴파일 검증, 단위 테스트, 동작 동등성 테스트, 사람의 코드 리뷰를 함께 수행해야 한다.

### 6.3 속도와 품질의 trade-off
실험 기록에 따르면 namotron3는 매우 빠른 속도를 보였지만 리팩토링 품질은 낮았다. 반대로 qwen25는 품질은 양호하지만 속도가 느렸다. gemma4는 품질과 속도 모두 양호한 모델로 관찰되었다. 이는 리팩토링 모델을 선택할 때 단순히 응답 속도만 볼 것이 아니라 테스트 통과율과 실패 유형을 함께 고려해야 함을 의미한다.

## 7. 결론
본 연구는 LLM 기반 코드 리팩토링의 가능성과 한계를 로컬 및 경량 모델 중심으로 검증하였다. Java 코드 스멜 세 유형에 대해 총 90개의 리팩토링 결과를 테스트한 결과, 전체 성공률은 64.4%였다. 모델별로는 gemma4가 76.7%로 가장 높았고, qwen25는 63.3%, namotron3는 53.3%였다.

코드 유형별로는 중복 코드와 긴 메서드에서 각각 86.7%, 80.0%의 높은 성공률을 보였지만, 거대 클래스는 26.7%로 낮았다. 따라서 LLM은 국소적이고 명확한 리팩토링에는 유용할 수 있으나, 복잡한 상태와 책임 분리가 필요한 구조적 리팩토링에는 아직 한계가 크다.

최종적으로 LLM 기반 자동 리팩토링을 실제 개발에 적용하려면 다음 조건이 필요하다. 첫째, public API와 기존 동작을 유지하라는 명확한 제약을 프롬프트에 포함해야 한다. 둘째, 리팩토링 결과를 반드시 테스트로 검증해야 한다. 셋째, 거대 클래스처럼 복잡한 리팩토링은 LLM 단독 자동화보다 개발자 검토를 포함한 보조 도구로 사용하는 것이 적절하다.

향후 연구에서는 샘플 수를 확대하고, 단순 테스트 통과 여부뿐 아니라 복잡도, 중복도, 메서드 길이 변화 같은 정적 품질 지표를 함께 측정할 필요가 있다. 또한 Agent 방식이나 하네스 엔지니어링을 적용하여 LLM이 임의로 API를 변경하지 못하도록 제약하는 방법을 추가로 검토할 수 있다.
